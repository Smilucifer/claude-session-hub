自由讨论 vs 投研圆桌

两种"多 AI 讨论"范式的本质差异 · 重构方向 · 三个落地选项 · 2026-04-28

⚡ 一句话 TL;DR

你的直觉大部分对、有一处偏差：

  • 对 投研圆桌的轮次状态机 / 圆桌卡片 UI / fanout-debate-summary 语义本来就是通用的，不是投研专属。
  • 对 它确实更轻、更适合"我有个 idea，三 AI 帮我一起想方案"这种场景。
  • 需修正 自由讨论 ≠ 编码场景。真正"默认编码"的是 driver-mode（主驾模式）。自由讨论现在已经是泛化的——只是 UI 是"三终端并排"，对话语义弱。
  • 不建议 "完全复用投研圆桌、只换 prompt"——投研版的 rules 写死了 LinDangAgent / 沙箱跑代码 / "纯读不写"等投研专属约束，硬搬到 PPT/数学/通信场景会污染。需要做泛化抽象，不是简单换字符串。

我的推荐方案：把 roundtable-orchestrator 提升成"通用圆桌引擎"，投研圆桌降级为它的一个 preset，自由讨论也用它做新的默认入口。详见末尾"③ 三个选项"。

③ 为什么投研圆桌的范式更适合"我有个 idea，三 AI 帮我一起想"

你描述的场景特征：

  • 用户有一个待解决的问题（不是要 AI 自己干活），希望 AI 当智囊。
  • 问题泛化：数学题、无线通信解决方案、PPT 思路、产品设计、写作大纲...
  • 讨论是多轮深化的：第一轮各自给思路 → 看到分歧 → 让某家综合。
  • 不是"看 AI 跑代码"——主要是读文字、想方案。

这套场景特征和投研圆桌几乎完美对齐：

✅ 圆桌的天然契合
  • fanout 隔离首轮：避免群体思维，三家本色出击
  • debate 中转：你能控制"什么时候让他们看对方观点"
  • summary 综合：让某一家做最终方案集成
  • 卡片 UI：直接看"三个观点 + 状态"，不用盯三个终端
  • covenant 可编辑：每个会议室定制讨论公约（场景+偏好+红线）
  • 本色发挥不扮演：你不用费脑想"谁演主驾谁演副驾"
⚠ 自由讨论的不契合
  • 三个 xterm 终端占满屏幕——但讨论场景你看的是文字，不是 CLI 输出过程
  • 无轮次概念——每条消息独立，多轮深化要自己手动维护上下文
  • SM 标记 + 黑板是为"协作做事"设计的（编码场景能用），讨论场景用不上
  • 需要主动 @ 路由——认知负担重
  • 无统一 prompt——三家各跑各的默认行为

所以你的判断"UI 应该借鉴投研圆桌"和"做轻一些"是正确的——这两点其实是同一件事的不同侧面。

⑤ 三个落地选项（请你选）

你的核心想法："完全复用投研圆桌、调 prompt"——这是其中一个选项（B）。但我推荐 A，原因写在每个选项后面。

A 推荐 把 roundtable 提升为通用圆桌引擎，投研降级为 preset，自由讨论默认接入

重命名 research-mode.js → roundtable-mode.js（或拆出 roundtable-core.js），把 covenant 抽象为场景模板库，新建会议时给一个下拉框： 

场景模板：
  ◯ 通用讨论（任意话题，无领域偏置）        ← 默认
  ◯ A 股投研（沿用现有 covenant + LinDangAgent 数据指引）
  ◯ 数学题（一题多解、严谨推导、互相 verify）
  ◯ 无线通信方案（标准/性能/复杂度三角权衡）
  ◯ PPT/写作（结构、叙事、视觉建议）
  ◯ 产品设计（用户/技术/商业三视角）
  ◯ 学术研究（文献定位、假设挑战、方法论）
  ◯ 自定义（粘贴你的公约文本）

自由讨论模式直接升级为"通用讨论"圆桌——用户进会议室默认就是圆桌 UI，发问就是 fanout，@debate/@summary 立即可用。

  • 优点：投研圆桌 = 通用圆桌的一个 preset；自由讨论也是。架构干净，任何新场景加个模板就行。
  • 优点：你提到的"做轻"自然达成——卡片 UI + 状态机本来就比三终端轻。
  • 优点：driverMode（编码主驾）保留，编码场景仍然有专属模式。
  • 代价：要改 ~3 个文件（research-mode.js / meeting-room.js renderer / 新建场景模板 JSON）。模板大约 150 行，重构估计 1-2 个会话。

为什么推荐 A：把"通用讨论"作为投研圆桌的兄弟而不是 fork，未来加场景零成本。投研版的 LinDangAgent 数据接入指引留在投研 preset 里不污染别处。

B "完全复用，只调 prompt"——你的原始想法

开新会议时让用户选 covenant 文本，自由讨论模式 = researchMode + 通用 covenant；不动其他代码。

  • 优点：实施最快（半小时内）。
  • 问题：rules 里的"数据获取（LinDangAgent）""纯读不写""沙箱跑代码"等条款会出现在 PPT/数学场景的 prompt 里——会引导 AI 做不相关的探查。
  • 问题：UI 标题硬编码"投研圆桌" / firstRunHint 文案 / 状态标签，要么硬留要么也得改——和 A 工作量差不多。
  • 问题：research-mode.js / researchMode / arena-prompts/*-research-mcp.json 命名都是"research"，长期看是技术债。

为什么不推荐 B："只调 prompt"听起来轻，实际执行后会发现 rules 里有大量投研专属约束，全删了 prompt 还能用？这是个伪轻量。要么硬扛污染，要么走到 A 的工作量但留下了乱命名。

C 双轨并存：自由讨论原样保留，投研圆桌单独存在，啥也不动

你想做编码协作就用自由讨论 + 黑板；想做投研就用圆桌；想做数学/PPT 就...自己开会议室手动敲。

  • 优点：零工作量。
  • 问题：你想的"轻量泛化场景"没有承载——要做数学题还得回到三终端 + 手动 @ 路由的旧模式。
  • 问题：圆桌 UI 的优势局限在投研一个场景，浪费。

为什么不推荐 C：违背了你这次提问的初衷（你说"我希望也做的轻量一些"）。

⑥ 风险与注意点

⚠ 真要走 A 方案，注意这几点

  1. 不要直接吃掉 driverMode。编码场景下"主驾 + 副驾审查"和"三家平等本色"是两种范式，前者效率更高（Claude 直接干，副驾只在关键点介入）。圆桌取代自由讨论，不取代主驾。
  2. covenant 需要"可空"语义。现在投研 covenant 是必填的，通用场景应该允许"不预设公约"——只 rules 一段，让模型本色发挥。
  3. 状态机最好支持"重置"。当前 roundtable.json 累积所有轮次，长会话上下文会爆。需要"clear/new turn-set"的入口。
  4. UI 终端折叠后，stream 输出谁来兜底。投研圆桌现在的卡片是"完整答案就绪后"渲染，中间过程不展示。如果 AI 在做工具调用（联网/读文件），过程信息要么折叠在卡片里，要么允许展开终端。这个设计要明确。
  5. resume 路径要重新测。research-mode 的 resume 注入了 RESEARCH_RESUME_REMINDER，通用化后这个 reminder 也要场景化。
  6. 命名兼容。已有的 arena-prompts/*-research.md / *-roundtable.json 文件 naming 既混用了 research 又用了 roundtable，重构时要拍板统一（建议都改 roundtable，研究场景另起 research preset 名）。

📋 A 方案最小可行步骤（如果你拍板）

  1. 新建 core\roundtable-presets\ 目录，每个场景一个 .md（covenant）+ 一个可选的 data-rules.md（场景专属数据指引，比如投研的 LinDangAgent 段）。
  2. 把 RESEARCH_RULES_TEMPLATE 拆成"通用 rules"+"投研 data-rules"两部分。
  3. 新建会议室时弹场景选择，写入 meeting.preset 字段。
  4. research-mode.js 重命名为 roundtable-mode.js；researchMode 字段改名为 roundtableMode（或保留向后兼容别名）。
  5. UI 文案动态化：标题"投研圆桌"→"{{preset.title}}圆桌"。
  6. 自由讨论模式默认改成"开会议 = 通用圆桌"；老的三终端 + 黑板布局降级为"经典模式"开关。

预估：spec + plan 一个会话，实施 1-2 个会话，E2E + 三路审查一个会话。总成本 ≈ 半天 ~ 一天。
