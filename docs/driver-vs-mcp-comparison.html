20 个维度逐一对比

1. 架构哲学
主驾模式
Hub 中心化。3 个 AI 是平等的"会议参与者"，Hub 是会议主持人。Claude 虽是主驾但被 Hub 调度。
CCB-MCP
Claude 中心化。Claude 是唯一的对话主体，Gemini/Codex 是 Claude 的"工具"。一切以 Claude 为中心。
本质差异：A 是"圆桌会议"模型，B 是"主管+助手"模型。
2. 会话状态模型
主驾模式
副驾是持久 PTY 会话，保留完整对话历史。
第二次问 Gemini 时它"记得"上次说过什么。
CCB-MCP
副驾是一次性进程（codex exec -）。
每次调用都是新会话，无记忆。需要 Claude 每次重新喂上下文。
影响：A 适合长会话探讨，B 适合短任务式问询。A 的 token 增量小（增量上下文），B 的 token 每次重置。
3. 用户交互范式
主驾模式
用户可以直接和任意 AI 对话：
· @gemini 你看这段代码 直接问 Gemini
· 切 tab 看每个 AI 的工作过程
· @review 是"群发审查"
CCB-MCP
用户只和 Claude 对话。
· "请让 Gemini 审查这段" → Claude 调用工具
· 副驾的输出由 Claude 转述给用户
· 用户看不到副驾的实际过程
影响：A 给用户"多 AI 团队"感，B 让用户感觉只有一个 AI 但能力强。
4. 可观察性
主驾模式
完全透明：每个 AI 在自己 tab 中实时可见。用户能看到 Gemini 思考、Codex 读哪些文件、Claude 怎么消化反馈。
CCB-MCP
黑盒：副驾的执行过程对用户不可见，只能看到 Claude 转述的结果摘要。出问题时难以排查。
对调试和信任的影响：A 适合需要"看到工作过程"的用户（如华为标准要求过程可追溯）；B 适合"只关心结果"的快任务。
5. Claude 的认知负担
主驾模式
Claude 专注代码工作，不负责编排。Hub 自动构建审查上下文，Claude 只需阅读副驾反馈。
缺点：Claude 不知道副驾"为什么这么说"，缺背景。
CCB-MCP
Claude 是编排者：要决定何时调用、传什么参数、怎么综合结果、是否再次调用。
认知负担更高，但 Claude 完全掌控审查逻辑。
权衡：A 让 Claude 更高效（认知带宽给代码），B 让 Claude 更智能（编排能力发挥）。
6. 触发机制
主驾模式
双触发：
· 用户主动 @review
· Claude 输出 [REQUEST_REVIEW] 标记自动触发
· 未来 MCP Tool 也可触发（计划中）
CCB-MCP
单触发：仅 Claude 主动调用 MCP 工具。
用户要审查必须告诉 Claude，Claude 决定是否调用副驾。
用户控制度：A 用户可绕过 Claude 直接调副驾，B 必须通过 Claude。
7. 可靠性
主驾模式
复杂的 PTY 链路：
· TranscriptTap 监听文件推断完成
· Gemini 多轮 tool use 误判
· conptyInheritCursor 时序问题
· PTY 写入丢失风险
CCB-MCP
标准 MCP 协议：
· 进程退出 = 完成（确定性）
· stdout JSON 解析，结构化
· 错误码明确
· 整体故障模式简单
工程风险：A 的 PTY 链路有 5+ 个潜在故障点，B 几乎只有"工具调用失败"一种。
8. Token 经济
主驾模式
副驾持续会话，每次审查只发增量上下文。
10 次审查总成本可能 = 1 次审查 × 1.5（前期摊薄）。
但长会话副驾 context 可能爆。
CCB-MCP
副驾每次从头加载整个 system prompt + 上下文。
10 次审查 = 10 倍单次成本。
但 token 上限可控，不会爆 context。
结论：长会话 A 省 token，短任务多次调用 B 更稳定。
9. 结果传播链路
主驾模式
Hub 收到副驾输出后多渠道分发：
· 渲染 Banner（用户即时看到）
· 写 timeline（永久记录）
· 写 .arena/reviews/*.md（Claude 可读）
· 通过 pendingReviewId 提示 Claude
CCB-MCP
单一渠道：tool result 直接给 Claude。
用户看不到原文，全靠 Claude 转述。
无独立持久化（除非 Claude 自己保存）。
权衡：A 信息丰富但分散，B 简洁但用户依赖 Claude 转述质量。
10. 故障模式
故障主驾模式CCB-MCP
副驾卡住PTY 等 turn-complete 永不返回进程超时退出，错误码返回
副驾输出格式错原文展示，Claude 自行理解JSON schema 校验
副驾问"yes/no"PTY 卡住等用户输入--full-auto 一次性流程不会问
多审查并发需全局锁天然隔离（不同进程）
结果丢失向 Claude PTY 推送可能被吞tool result 直返不可能丢
11. 扩展性
主驾模式
加新副驾 = 加新 PTY kind + 适配 TranscriptTap + UI tab + ...
但可加角色（character）、记忆、自定义 prompt 文件。
CCB-MCP
加新副驾 = 加新 MCP 工具（一个函数）。
10 行代码搞定，但功能受限于工具签名。
结论：MCP 简单但扁平，主驾复杂但深度。
12. 多 AI 协作模式可能性
主驾模式
可以演化出复杂协作模式：
· Gemini 和 Codex 互相对话（@gemini 看了 @codex 的回复后...）
· 角色扮演（Gemini 当架构师 PK Codex 当工程师）
· 多副驾辩论模式
CCB-MCP
始终是单方调用：Claude 调 → 副驾答 → Claude 综合。
无法实现副驾互相讨论。
Claude 是唯一的"信息中转"。
关键洞察：A 有"会议室"潜力，B 永远是"个人助手 + 工具"。
13. 学习曲线
主驾模式
用户需要理解：会议室、tab、@review、@gemini、@codex、driver mode、blackboard...
新概念多，但学会后控制力强。
CCB-MCP
用户只需对话："请让 Gemini 帮我看看"。
符合直觉，零学习成本。
用户画像：A 适合 power user，B 适合普通用户。
14. 开发复杂度
组件主驾模式CCB-MCP
核心代码量~3000 行~300 行
需要的技术栈Electron, PTY, xterm, IPC, TranscriptTap, fs.watchMCP SDK, child_process
故障排查难度高（多层抽象）低（线性流程）
测试难度需要 CDP + 真实 CLI单元测试可覆盖
15. 数据持久化
主驾模式
多层持久化：
· Hub state.json（会议元数据）
· meetings/{id}.json（timeline）
· .arena/context.md（快照）
· .arena/reviews/*.md（每次审查存档）
· 副驾 transcript（CLI 自带）
CCB-MCP
单层持久化：仅 Claude transcript。
副驾输出仅在某次 tool result 里出现，难以独立检索。
除非 Claude 主动保存。
审计/回溯：A 完整可追溯，B 依赖 Claude transcript 检索。
16. 危险操作控制
主驾模式
副驾在 PTY 中可能失控（如刚才 Codex 想跑测试服务器）。
需要在 prompt 中约束 + sandbox 配置 + 权限提示拦截。
CCB-MCP
MCP 工具签名定义清楚，输入输出受控。
副驾一次性进程 + sandbox，无法持续做坏事。
安全性：B 天然更安全（短生命周期 + 函数签名）。
17. 中断恢复
主驾模式
Hub 重启后会议室状态可恢复：
· dormant 子会话可 resume
· timeline 持久化在磁盘
· pendingReviewId 持久化
CCB-MCP
Claude 进程重启 → MCP 重连 → 历史调用记录在 transcript 里。
副驾本就一次性，无状态可恢复。
用户体验：A 重启后能"接着开会"，B 重启就重置。
18. 适用场景
场景更适合原因
长时间编程会话（4h+）主驾持久会话 + 持续观察 + token 摊薄
快速代码审查（5 min）MCP低门槛 + 简单可靠
多 AI 辩论/共识主驾副驾互可见 + 圆桌结构
研究探索主驾透明可观察 + 可深入互动
执行明确任务MCP编排集中 + 结果聚合
团队协作演示主驾3 个 tab 视觉化展示
移动端使用MCPHub 是桌面应用，MCP 在 CLI 层
CI/CD 自动审查MCP无需 GUI，纯 CLI 流程
19. 混合可能性
两者其实可以共存。当前 Hub 计划的 B4（MCP Tool）就是在主驾模式中嵌入 MCP 能力：
  • Hub 仍然管理 3 个持久 PTY 会话（保留观察性、协作能力）
  • 同时给 Claude 暴露 MCP tool（Claude 也可以编排式调用）
  • Hub 的 executeReview() 是"统一执行层"，被两种触发方式复用
这样既有"会议室"的多 AI 协作感，又有 MCP 的结构化可靠性。
缺点：开发复杂度更高（两套触发链路 + 共享执行层）。
20. 战略定位
主驾模式
"AI 团队工作站"。把 AI 当作可观察的同事，用户是会议组织者，每个 AI 有独立空间和角色。
远期可演化为 cat-cafe 那样的多 AI 角色系统。
CCB-MCP
"超能 Claude"。Claude 通过工具变得无所不能，Gemini/Codex 是它的能力扩展。
远期可加更多 MCP tool（搜索、画图、数据库等），但永远是"Claude + 工具"。
哲学差异：A 相信"多 AI 协作 > 单 AI 强化"，B 相信"单 AI + 工具堆叠"。

两种典型场景对比

场景 1：审查 auth.js 修复（5 分钟任务）
主驾模式：用户输入 @review → Hub 自动构建上下文 → 并行发给 Gemini+Codex → 用户看 banner 知道结果 → 切 tab 看详细分析 → @claude 让主驾处理 FLAG → 完成
步骤：5-6 步 | 用户操作：3 次

CCB-MCP：用户告诉 Claude "请让 Gemini 和 Codex 审查刚才的修复" → Claude 调 ask_gemini → 调 ask_codex → 综合给用户文字总结 → 用户决定是否让 Claude 修 → 完成
步骤：5-6 步 | 用户操作：2 次（更少）
场景 2：4 小时深度重构会话
主驾模式：多次 @review、副驾保留上下文（"还记得我们讨论的 OAuth 设计吗？"）、用户偶尔 @gemini 单独探讨架构、最后 @review 总结 → token 成本随会话增长慢

CCB-MCP：多次让 Claude 调用副驾、每次副驾从头加载（重新发完整 system prompt + 当前 context）、副驾不记得之前的讨论 → token 成本线性增长，4h 后可能爆

差距：长会话场景 A 省约 60% token + 体验更连续。
场景 3：用户想看 Gemini 怎么"思考"的
主驾模式：切到 Gemini tab → 实时看到 Gemini 读哪些文件、推理过程、tool use → 完美透明

CCB-MCP：Claude 调用工具后只能看到 Gemini 的最终输出（stdout），过程被 stdout pipe 屏蔽。要看过程只能加 stderr 日志，但和正常输出混在一起难解析。

差距：A 完胜，B 几乎不可能做到这点。

综合结论
主驾模式是"产品"，CCB-MCP 是"工具"。

选 A 的理由：你想做一个真正的 AI 团队工作站，强调过程可观察、多 AI 平等协作、长会话连续性。这是一种新型工作范式，类似从"自己写代码"到"管理一个 AI 团队"的认知升级。

选 B 的理由：你想给 Claude 加点"外挂"，让它在需要时调用其他 AI。本质上 Claude 仍是唯一的对话伙伴，只是变得更强。开发简单、可靠、易维护。

当前 Hub 的方向已经是 A + B 的混合：保留主驾模式的 PTY 会议室架构（A），同时计划加 MCP Tool（B4）让 Claude 也能主动触发审查。这种混合方案：
  • 保留 A 的核心价值：可观察、多 AI 协作、持久会话
  • 用 B 的可靠性补强 Claude 自触发链路（避免 [REQUEST_REVIEW] 文本标记的脆弱性）
  • 缺点是工程复杂度最高（既要维护 PTY 链路又要维护 MCP server）

是否值得？取决于你长期的产品方向。如果是个人深度编程工具—— A 方向更对。如果是通用 AI 工作流—— B 更普适。
