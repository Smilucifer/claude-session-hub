1 · 项目定位

以前开多个 PowerShell 窗口跑多个 Claude Code 会话，窗口到处飞、切换靠 Alt+Tab、有没有新回复只能一个个点。这个项目把多会话集中到一个 Electron 窗口里，左侧像微信一样列会话，未读数、最新问题预览、状态指示一目了然，右侧是聚焦会话的终端。


用 Electron 而不是浏览器端的原因：需要直接管 PTY，浏览器端要走 WebSocket + server，多一层复杂度（这也是 v0 架构，已迁移）。

6 · 关键技术决策

1
为什么用 Electron 而不是继续浏览器 + 本地 server？
直接在进程内管 PTY、读剪贴板、写本地文件，不用 WebSocket + Express + 跨进程鉴权。分发用 Electron installer 而不是 npx。
Why：功能越做越偏 OS 层（剪贴板、文件、窗口焦点），浏览器沙盒越来越碍事
2
为什么不用 React / TypeScript？
renderer 只有 ~500 行，交互模式简单（列表 + 单终端面板），原生 DOM + CSS 变量足够。无构建步骤 → 改代码直接刷新。
Why：Simplicity first；引入 React + Vite 不解决任何实际问题，只增加复杂度
3
为什么放弃 Hook 驱动，改回 buffer 读取？
Hook 方案依赖 ~/.claude/settings.json 里全局 hook 注册，用户机器环境不稳定；而 buffer 内容已经在 xterm 里渲染好，直接读成本更低。
Why：hook 链路 = PTY → Claude Code → Python hook 脚本 → HTTP POST → Hub server；中间任何一环挂掉就失灵。buffer 读取是内存内操作，可靠性高一个数量级
4
为什么图片粘贴要自己做文件落盘，而不是传 base64？
Claude Code CLI 接受的是文件路径（它自己读 PNG），不是 base64 数据流。我们的工作就是复现微信曾经做的事：剪贴板 bitmap → 磁盘文件 → 路径。
Why：这是 Claude Code CLI 的输入契约，不是我们的选择

7 · 踩过的坑（图片粘贴三层 Bug 连环）

#1

paste 事件注册在 bubble 阶段，被 xterm 吞了

第一版用 document.addEventListener('paste', fn)（默认 bubble），合成事件能触发，但真 Ctrl+V 什么都不响应。原因：xterm.js 内部在 .xterm-helper-textarea 上的 paste handler 调用了 stopPropagation()，bubble 阶段的 document listener 永远收不到。
改成 capture 阶段：addEventListener('paste', fn, true) + stopImmediatePropagation()
#2

Claude Code 需要 bracketed paste 包装

第二版直接 ipcRenderer.send('terminal-input', path) 写到 PTY，PowerShell 会话下字符能显示，但 Claude Code 会话下彻底没反应。原因：Claude Code 开了 bracketedPasteMode，裸字符串被当成键盘输入丢弃；它期望 \x1b[200~<path>\x1b[201~。
改用 xterm 自己的 terminal.paste(text)，它会自动按模式包 escape 序列
#3

Chromium 原生 Ctrl+V 在 xterm textarea 上不触发 paste 事件 [致命]

前两层修完后自动化测试全绿：webContents.paste() 能触发事件链、[Image #1] 能出现在 Claude Code 输入框。但用户真键盘 Ctrl+V 依然失败。
真相：webContents.paste() 和真键盘 Ctrl+V 是两条不同的代码路径。前者直接给 focused element 发 paste 事件；后者是 Chromium 的 native paste action，对 xterm 的 hidden helper textarea 这种特殊元素不发 paste 事件。诊断日志：5 次 keydown 全部捕获，0 次 paste 事件。
彻底放弃 paste 事件，改用 keydown 钩子拦截 Ctrl/Cmd+V（capture 阶段）

💡 核心教训（已记入 auto-memory）

自动化触发 ≠ 用户真操作。 webContents.paste() / dispatchEvent / execCommand('paste') 这些模拟 API 走的是和真键盘完全不同的代码路径。用前者测通不代表用户场景能用。涉及键盘/鼠标/剪贴板类 OS 层交互，要么装诊断钩子让真实操作触发验证，要么明确标注"未覆盖真实用户路径"，不得声称端到端验证通过。
