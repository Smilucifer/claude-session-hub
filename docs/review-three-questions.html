Q1: 你选的方案 C 改良版 — 下次交互时显式提示 Claude 查看

你的思路：审查结果不推送到 Claude PTY，而是写入文件。下次用户给 Claude 发消息时，Hub 在消息前面自动注入一段提示，让 Claude 主动去读。

具体实现方案

1副驾完成审查 → 结果写入 .arena/review-result.md + timeline
2Hub 在内存中标记 meeting.pendingReview = true
3用户下次向 Claude 发消息（普通文本 / 任何输入）
4Hub 检测到 pendingReview = true → 在用户消息前自动注入：
[系统提示] 副驾刚完成审查，结果已写入 .arena/review-result.md，请先读取并处理。 [用户原始消息] 继续修下一个 bug
5Claude 收到后自然会先 Read .arena/review-result.md → 处理反馈 → 再做用户的事
6Hub 重置 pendingReview = false
为什么这比纯 rules.md 靠谱：
  • rules.md 是 system prompt 中的软约束，Claude 可能忽略
  • 注入到用户消息前面 = 显式出现在 Claude 的输入流中，Claude 不可能忽视
  • 类似于 cli-caller 中 Claude 主动读输出，但由 Hub 触发时机

注入位置的实现

在 handleMeetingSend() 的 driver mode 分支中，发送给 Claude 之前检查： 
// meeting-room.js handleMeetingSend(), driver路径发给Claude前
if (current.pendingReview) {
  const hint = '[系统提示] 副驾刚完成审查，结果已写入 .arena/review-result.md，请先读取并处理。\n';
  payload = hint + payload;
  current.pendingReview = false;
  // 更新持久化状态
}

Q2: Gemini 多轮 tool use 的影响 — 确认是 Bug

问题重现

Gemini 审查时的典型行为：
Gemini 收到审查 prompt
Gemini 决定先看看代码 → 调用 tool: read_file("auth.js")
工具返回文件内容 → Gemini 生成第一轮回复（分析中...）
← GeminiTap 此时触发 turn-complete（type:"gemini" + tokens.total 出现）
text = "让我先看看 auth.js 的代码..."（不完整！）
Gemini 继续 → 调用 tool: read_file("auth.test.js")
工具返回 → Gemini 生成最终回复（完整审查意见）
← GeminiTap 再次触发 turn-complete
text = "OK: 修复合理，覆盖了主要路径..."（完整）

当前代码行为

// main.js driver-request-review
const handler = (ev) => {
  if (ev.hubSessionId !== sid) return;
  transcriptTap.removeListener('turn-complete', handler);  // ← 只听一次
  resolve({ sid, ...driverMode.parseReviewVerdict(ev.text) });    // ← 用第一次的 text
};
transcriptTap.on('turn-complete', handler);
Bug：第一次 turn-complete 就 resolve + removeListener → 拿到的是 "让我先看看..." 而非最终审查意见。
后续的真正审查结果 → 事件被丢弃，无人消费。

影响评估

副驾 是否受影响 原因
Gemini 是 — 多轮 tool use 时截断 GeminiTap 的 turn-complete 在每轮 type:"gemini" 输出时触发，非仅最终轮
Codex 否 CodexTap 仅在 task_complete 事件触发，这是 Codex 的真正完成信号
Claude 否（不参与审查） ClaudeTap 通过 Stop hook 触发，也是真正完成信号

修复方案

方案 X: Debounce — 等稳定后再取

不在第一次 turn-complete 就 resolve。
每次 turn-complete 重置一个 debounce 计时器（如 5s）。
5s 内没有新 turn-complete → 认为真正完成 → resolve 用最新 text。

简单 增加 5s 延迟

方案 Y: 持续更新 — 最终取最后一次

每次 turn-complete 都更新暂存的 text（不 resolve）。
同时启动 debounce 计时器（3-5s）。
连续 N 秒无新事件 → resolve 用最新暂存的 text。

效果相同，本质和方案 X 一样，只是明确"暂存最后一次"。

可靠 同样延迟
延迟可以接受吗？
Gemini 多轮 tool use 之间的间隔通常 2-10s（读文件 + 思考）。
5s debounce 意味着 Gemini 最后一轮输出后再等 5s → 总额外延迟 5s。
对于一个本身就要 20-60s 的审查流程，5s 完全可接受。

对 Codex 无影响：Codex 只有一次 turn-complete（task_complete），debounce 到期后立即 resolve。
