3 差距总结 — 硬缺口 / 软差距 / 超越点
P0 杰尼龟零 MCP 工具
Codex CLI 原生支持 MCP（TOML 配置），但 Hub 集成路径完全没注册 MCP server。杰尼龟只能通过 persona 文件接收指令，无法主动调用 team 工具（recall_facts / write_fact / team_respond）。这导致杰尼龟是"只读角色"——回答靠 stdout 解析，无法参与知识写入和协作。

修复路径：参照皮卡丘的 _writeMcpConfig()，为 Codex 生成 TOML MCP 配置，传入 exec -c mcp.servers。预计 +25% 等效度提升（40%→65%+）。
P0 无 Session Chain / Bootstrap
三角色在 room close 时都丢失上下文（PTY 被杀、ACP 关闭、Codex SID 悬空）。cat-cafe 的 session chain 实现了 seal→digest→bootstrap→new session 的无缝接力，即使 CLI 重启也能通过 2000 tok 摘要恢复关键上下文。

修复路径：在 team-session-manager.js 实现 SessionSealer（seal 触发条件：context health 85% / 手动 / 超时），将 digest 写入 team.db sessions 表，新 session 启动时 bootstrap 注入。
P0 无 Intent 路由 / 并行思考
当前 Hub 只能串行 @mention 一个角色。cat-cafe 实现了 ideate（并行独立思考）/ execute（串行协作）两种模式，由 IntentParser 自动判断。指挥部战术（多人独立分析再碰头）正是 ideate 模式。

修复路径：在 sendMessage 前加 IntentParser（正则 + 关键词 ideate/execute），ideate 模式 Promise.all 并行 spawn 三角色，收集结果后合并展示。
P1 跨会话记忆仅 FTS，无向量搜索
当前 MCP 的 recall_facts / search_facts 基于关键词（FTS5 + LIKE），语义相近但词汇不同的知识检索不到。cat-cafe 用 sqlite-vec + all-mpnet-base-v2 (384 维) 实现语义搜索 + rerank。

修复路径：在 Python MCP server 的 memory 模块加 EmbeddingService（可用 sentence-transformers 本地模型），team.db 新增 vec_facts 表。预计 +10% 记忆维度提升。
P1 小火龙 ACP 无自动重连
AcpClient 进程意外退出后，不会自动重建 session。下次 sendMessage 才发现连接断了，用户体验为"失败一次后恢复"。

修复路径：监听 AcpClient.proc 的 close 事件，延迟 2s 后自动 start() + initialize()。在新 session 中注入上一个 session 的摘要（需先实现 Session Chain）。
P1 Gemini ACP 路径不写 team.db
Code review 发现的 C1 问题：Claude 通过 team_respond MCP 工具写 team.db，Codex 通过 Hub 解析 JSON 后写入，但 Gemini ACP 路径的 _sendGeminiAcpMessage 没有调用 _writeTeamDbEvent。造成持久化不对称——Gemini 的回复不进入事件流。

修复路径：在 _sendGeminiAcpMessage 的 return 前加 this._writeTeamDbEvent(roomId, characterId, content)，一行代码。
P2 无跨家族自动 Review
cat-cafe 的 ReviewPolicy + reviewer-matcher 实现了"不同家族优先 review"的自动分配。当前 Hub 只有手动 @mention 触发 review。

修复路径：移植 reviewer-matcher 逻辑到 team-session-manager.js，在代码类消息后自动触发跨角色 review。
P2 无知识生命周期管理
cat-cafe 的知识有 L0→L4 成熟度分级（Episode→Pattern→Draft→Validated→Standard），量化晋升阈值（≥6 次引用 + ≥2 agent + ≥80% 成功率）。当前 team.db 的 facts 没有成熟度追踪。

修复路径：facts 表增加 maturity_level / use_count / success_rate 字段，MCP reflect 工具在反思时自动评估晋升。
✅ 已领先 — Route A+ 保留原生能力
cat-cafe 和原生 CLI 之间有 persona 注入 + 工具限制的 overhead。我的 Route A+ 设计让每个角色 = 完整 CLI session + 最小系统 prompt (~600 tok)，原生 Skills / Hooks / Plan Mode 天然可用。这是 cat-cafe 做不到的——它必须通过 MCP 适配层间接访问 CLI 能力。
✅ 已领先 — Persona 零 overhead
2026-04-21 人格简化后，三角色 YAML 从 65 行降至 12 行，persona 字段全空，模型以原生能力裸露发挥。cat-cafe 的布偶猫/暹罗猫/缅因猫仍有 personality / voice / strengths / caution 等多字段 overhead。我的 token 效率更高。
✅ 已领先 — Codex exec resume 原生链
杰尼龟的 exec resume 保留了完整 Codex JSONL rollout 语义，session ID 跨消息持久化，恢复时 Codex CLI 重建全部上下文（system messages + tool calls + reasoning）。cat-cafe 的缅因猫使用通用 session chain 抽象，不利用 Codex 原生的 resume 能力。
✅ 已领先 — ACP 现代协议
小火龙的 ACP (Agent Client Protocol) 是 Gemini CLI 的 IDE 级接口，JSON-RPC 2.0 + 反向请求 MCP 工具注册，比 cat-cafe 的通用 CLI spawn 更稳定、更可扩展，天然支持流式思考事件。
4 超越路线 — 在两个维度做出单 CLI 不可能的事

🎯 超越点 A：多视角交叉验证（Multi-Perspective Cross-Validation）

单个 Claude/Gemini/Codex session 只有一个视角。三角色并行分析同一问题，天然产生多样化观点——Claude 擅长架构全局、Gemini 擅长创意发散、Codex 擅长安全审查。当三者结论一致时可信度极高，当三者分歧时恰好暴露盲区。这是单 CLI session 永远做不到的。

收益：分析可信度 > 任何单模型，已被指挥部战术验证

实现路径：实现 ideate 模式 → 三角色并行分析 → 自动提取共识/分歧 → 合成报告。参照 cat-cafe Mode B 的 6 阶段流程（独立思考→冲突讨论→综合→Review→CVO 决策），但更轻量——仅在分歧时触发讨论。

🎯 超越点 B：共享知识蒸馏（Shared Knowledge Distillation）

三角色共享 team.db，一个角色发现的事实（如"该股近期主力净流出"）自动对其他角色可见。当加上向量搜索 + 知识生命周期后，team 的集体知识 > 任何单 CLI session 的局部记忆。这是1+1+1 > 3 的核心机制。

收益：集体智慧涌现，知识不再随 session 消亡

实现路径：① 补齐杰尼龟 MCP → 三角色均可写知识 ② 加向量搜索 → 语义检索 ③ 加 maturity 追踪 → 高频引用的知识自动晋升为"team 共识" ④ 新 session bootstrap 注入 top-K 高成熟度知识。

💡 差异化方向 C：Route A+ 的原生能力透传

cat-cafe 通过统一抽象层屏蔽了 CLI 差异，但也丧失了各 CLI 的独特能力（Claude Skills、Codex sandbox approval、Gemini multimodal）。Route A+ 保留完整原生能力 = 每个角色同时是 Team member + 独立 CLI expert。

收益：任何 CLI 新功能（如 Claude 新 skill）自动对 Team 可用，零适配成本

💡 差异化方向 D：轻量治理 vs 重型框架

cat-cafe 是完整的 Fastify + Redis + SQLite + MCP + WebSocket 栈，部署门槛高。我的 Team Room 嵌入 Hub（Electron）内，零额外依赖，一键启动。对个人用户而言运维成本差两个数量级。

收益：零部署 / 零运维 / 桌面即用
