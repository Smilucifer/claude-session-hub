总览：为什么是双轨

圆桌想做"无限自学习疯狂进步"，核心矛盾是 train-test distribution shift：

  • 禁用联网回测 → AI 能力被阉割，学到的不是真本事
  • 开放联网回测 → AI 能从搜索引擎拿到"未来日期"信息作弊

双轨方案把这个矛盾显式拆开，让两轨各自训练 AI 不同维度的能力：

🟢 轨道 1 · 量化训练

关闭三家所有联网/沙箱/MCP，只给硬数据（K 线/财报/资金流）+ 去标识化（股票名→代号）。

训练能力纯量化盘感（看 K 线判方向、看资金流判主力意图、看大盘节奏）
样本规模夜间 30-50 案例
样本来源历史时点（2018-2022 远古区）
作弊风险极低（去标识化 + 无联网）
判卷T+10 涨跌（DeepSeek 抽方向）

🟣 轨道 2 · 自然 self-play

实战即训练。三家全开联网/沙箱/MCP，你白天真实场景讨论，T+10 后自动判卷。

训练能力综合能力（信息整合、多源交叉、实时判断、新闻面）
样本规模每周 5-15 案例
样本来源用户真实讨论的票
作弊风险零（真实场景就是真实场景）
判卷T+10 涨跌（同上）

🔵 共用引擎

两轨产出的反思汇入同一引擎：三家互审 proposals → covenant/profile/prompt staging → 双轨 holdout A/B 验证 → 两轨都 +3% 才采纳（防偏科）。

🟢 轨道 1 · 量化训练 详细设计

核心理念

把"作弊"消灭在数据源：让 AI 看到的不是"兆易创新 / 2024-08-15 / 96.5"（脑子里有印象），而是"STOCK_T001 / D-0 / close=96.5 / industry=IND_A"。LLM weight 里没有 STOCK_T001 的记忆。

🔑 关键设计逻辑：去标识化让 LLM 的"权重作弊"失效，剩下的就是"看数字判方向"的纯量化能力——这正是我们要训练的。

触发时机

触发方式说明默认参数
A 用户手动Hub UI "投研圆桌" 旁加按钮 "量化训练"，点击弹"跑多少案例 / 远古日期窗口"30 案例 / 2018-2022
B 夜间 cron每晚 23:30 启动，跑 30 案例后自动停（避免占用主进程）30 案例 / 滚动日期
C 累积反思达 30 案例自动触发"共用反思引擎"阈值 30

完整流程（单案例）

1选题
LinDangAgent 提供股票池（按市值分层 + 过滤 ST/停牌 + 流动性≥5000 万）
随机抽 1 只 + 1 个时点（exam_date 在 [2018-01-01, 2022-12-31] 区间内随机）
生成元组：(ts_code, exam_date)
2数据预拉
Hub 调 LinDangAgent 新增 endpoint fetch_for_quant_arena（time_lock=exam_date）：
  • K 线（D-90 到 D-0，含 OHLCV + 量比 + 换手率）
  • 财报最近 4 期（营收 / 净利润 / 现金流 / 负债率）
  • 资金流（北向 / 龙虎榜 / 主力净流入近 20 日）
  • 大盘指数 K 线（沪深 300 同期）
  • 板块指数 K 线（行业板块同期）
  • 不拉 新闻 / 研报 / 舆情（轨 1 训练纯量化）
3去标识化（防作弊核心）
新建 core/quant-anonymizer.js，做以下替换：
原始替换后说明
兆易创新 / 603986STOCK_T001每个案例独立编号
半导体板块INDUSTRY_A但保留"防御 vs 进攻"二元标签（决策必需）
2024-08-15D-0所有日期改成相对天数
2024-08-14D-1K 线序列：D-90, D-89, …, D-1, D-0
沪深 300 / 创业板指MARKET_INDEX_M / GROWTH_INDEX_G同样匿名化
映射表 {stock_code → STOCK_T001, exam_date → D0} 存到 quant_sessions/<sid>/mapping.json，判卷时反查。
4创建去标识化圆桌会议
新会议室 mode='quant-training'（与 driver/research 平级新模式）：
  • system prompt 顶部加 训练公约："本次为量化训练。禁止调用任何外部工具（WebFetch/WebSearch/Bash/MCP/sandbox 全部）。所有判断只能基于 prompt 中的数据。违规会被审计。"
  • 注入预拉的去标识化数据包
  • 注入用户 covenant（让 AI 仍按你的风格判断）
  • 三家 spawn（关闭对应 CLI 的工具开关：Claude --allowed-tools=""、Gemini 不传 MCP、Codex 不传 MCP）
5圆桌运作（fanout → debate → summary）
与现有圆桌完全一致，但 summary 阶段强制结构化尾标：
讨论自由展开 ... 1500 字 ...

<<TENDENCY: bullish | bearish | neutral, confidence=0.0-1.0>>
判卷优先用尾标，没有时让 DeepSeek 抽方向。
6工具违规监控（反作弊审计）
Hub 监听 transcript_tap，扫每家 transcript 中的 tool_use 块：
  • 检测到 WebFetch / WebSearch / Bash / mcp__* 调用 → 案例标 contaminated=true
  • contaminated 案例 不进训练集，但保留供事后排查
  • 累积污染率 > 10% 触发告警，可能 prompt 约束失效
7T+10 解锁 + 判卷
解锁 exam_date + 10 个交易日的实际涨跌：
  • 个股 α = 实际收益 - max(大盘 β, 板块 β)
  • 看多 + α > 0 → hit；看空 + α < 0 → hit；中性 + |α| < 3% → hit
  • 三家分别记 hit/miss，summary 那家额外算"决策方"hit/miss
8落盘
写 ~/.arena-research/quant_sessions/<sid>.jsonl：
{
  "sid": "uuid",
  "exam_date": "2024-08-15",
  "ts_code": "603986.SH",
  "anonymized_as": "STOCK_T001",
  "industry_real": "半导体",
  "industry_anon": "INDUSTRY_A",

    "gemini": "neutral",

  "hits": { "claude": true, "gemini": true, "codex": false, "decision": true },
  "contaminated": false,
  "transcript_paths": { "claude": "...", "gemini": "...", "codex": "..." }

轨道 1 关键 issue（10 条）

Q1去标识化反作弊有效性需要实测
去标识化是假设 AI 无法从 K 线形态联想到具体票，但理论上 AI 可能识别"2007 年某月发生过的 4 个一字板涨停 + 解禁"这类罕见模式。 必须先跑 10 个测试案例，hit_rate 显著高于 53%（如 80%+）说明仍作弊，需要更激进的匿名化（如随机化 K 线日期顺序但保留相对位置）。
Q2行业匿名化对决策能力的影响
完全隐去行业（"半导体" → "INDUSTRY_A"）AI 学不到"防御性消费 vs 高弹性科技"的不同决策。
方案：保留二元标签（"防御型 vs 进攻型"，"周期 vs 成长"），但隐去具体行业名。这是关键的 trade-off 设计点。
Q3大盘指数也必须匿名化
沪深 300 是公开信息，AI 看到 "2008-10 沪深 300 走势" 立刻能猜出"金融危机"。
方案：大盘 K 线也只给相对值（基准化为 100），不给绝对点位；指数名替换为 MARKET_INDEX_M。
Q4三家 CLI 工具禁用机制不一致
Claude 有 --disallowed-tools；Gemini / Codex 不一定有同等 fine-grained 开关。
方案：① 不传 MCP config（断 MCP 通道）；② prompt 强约束 + transcript 审计兜底；③ 实测三家是否都遵守 prompt 约束。
风险：Codex 默认开 web_search 工具，必须找到关闭方法或 acknowledged 案例污染。
Q5样本日期需避开三家 cutoff
Claude 4.7 cutoff 2026-01；Gemini 2.5 cutoff 较早；Codex 不同。
方案：取三家 cutoff 最早的那个作为硬上限；2018-2022 是安全区。
但：2018 之前数据量大但市场结构变化大（注册制、北向通、科创板都没有），样本意义递减。
Q6轨 1 跑成本
单案例：3 家 × （fanout 1 + debate 1 + summary 0.33）= ~7 次 LLM 调用。
30 案例/晚 × 7 次 = 210 次/晚。Opus 占大头，月成本估算 $200-500。
方案：轨 1 默认用 Sonnet 跑（成本低 + 模拟 LinDangAgent 战绩 53%），Opus 只用于反思阶段。
Q7Opus 反思 prompt 长度爆炸
30 案例 × 三家 outputs × 平均 1500 tok = 135K tok，超 Opus 单次上下文。
方案：分批反思（10 案例一批，每批一次 Opus 提 mini-proposals），最后一次 meta-reflection 汇总。
Q8LinDangAgent fetch_for_arena 未必支持远古日期
现有 fetch 接口默认拉最近 N 天数据。需要扩展 time_lock 参数贯穿整个数据获取链。
方案：复用 build_report_context(time_lock=...) 已有逻辑，新增 fetch_for_quant_arena wrapper 输出去标识化前的 raw JSON。
Q9结构化尾标 vs prompt 反向强化
之前已被你纠正过 "prompt 不要用反例字串" — 这次设计 <<TENDENCY: bullish | bearish | neutral>> 时，要避免把"这种格式 OK / 那种格式不行"反例排比写出。
方案：只描述应该的样子（"在 summary 末尾输出一行尾标，格式：<<TENDENCY: 三选一, confidence=0-1>>"），不展示反例。
Q10去标识化对话本身的工程量
LinDangAgent 不太可能现成支持。需要：① 新建 core/quant-anonymizer.js（~150 行）；② LinDangAgent 加 fetch_for_quant_arena.py（~80 行）；③ 主驾/research mode 之外新增 quant-training-mode.js（~120 行）。

🟣 轨道 2 · 自然 self-play 详细设计

核心理念

实战即训练。零 distribution shift——你白天怎么用，AI 学到的就是怎么用得好。零作弊——真实场景没有"未来信息"可作弊。

🔑 关键代价：样本积累慢（一周 5-15 案例），第一次反思要等 4-6 周；改进速度比轨 1 慢一个量级。

轨道 2 关键 issue（8 条）

N1真实场景判卷的 selection bias
你白天选讨论的票不是随机抽样——你只挑感兴趣的（往往是已经在动的票）。
样本天然偏向"高 momentum 标的"，反思学到的"经验"可能是"在用户感兴趣的高动量股票上要怎么做"，不是普遍规律。
缓解：反思 prompt 里明示这点，让 Opus 区分"用户偏好场景的局部经验" vs "普遍规律"。
N2自由讨论 vs 强制方向标的冲突
你之前说"问题想清楚就够了，不必非要给决策"。如果 summary 没明确给方向，DeepSeek 抽不出怎么办？
方案：① 把"是否给方向"作为讨论性质的标签（决策类讨论 vs 思辨类讨论），只决策类进 outcome；② 思辨类讨论不进训练，但仍归档供层 1 用户画像更新。
N3联网信息的 attribution 难题
AI 在真实场景看了某条新闻才判断对，反思时新闻已是"历史"。
问题：反思学到的是"判断好"还是"看到那条新闻好"？反思 prompt 区分不清就学错方向。
方案：transcript 里保留 AI 调用的所有工具结果（已有），反思时让 Opus 看到"AI 看了哪些信息源 → 怎么判断"，attribution 才精确。
N4covenant 版本演化导致样本不可比
样本积累过程中 covenant 改了多次（共用引擎采纳 proposals）。某个旧样本是"covenant v1 下的判断"，新样本是"v3 下"，hit_rate 直接平均没意义。
方案：每个 nat_session 记 covenant_snapshot_hash，反思时按版本聚合统计。或采纳 proposal 后清空当前桶（损失旧样本但保证统计纯度）。
N5样本量 vs 反思频率的现实矛盾
一周 5-15 案例 → 月度 30-60 → 第一次反思要等 4-6 周。
早期阶段反思的统计意义弱（30 案例 hit_rate 标准差 ~9%，看不出真信号）。
方案：① 早期与轨 1 合并触发反思（把统计意义凑足）；② 阈值动态：前 3 个月用 60 案例阈值，之后降到 30。
N6复权问题
判卷时拉 T+10 涨跌，如果中间发生分红 / 送股 / 拆股，前后复权选哪种？
方案：统一用前复权（A 股惯例），与 LinDangAgent 一致。
风险：除权除息日附近的判卷可能偏差大，特殊场景需要 acknowledge。
N7用户讨论中途打断或换话题
你可能讨论一半就走了，没到 summary 阶段。这场讨论 outcome 怎么办？
方案：只有显式触发 @summary @<who> 的会议进入 outcome 系统；fanout/debate 阶段中断的不进训练。
N8真实场景 outcomes.db 整合
LinDangAgent 已有 outcomes.db（给 war_room 用）。圆桌的 nat_session 应该写到同一个 db 还是独立？
方案：独立（schema 不同 + 圆桌特有字段如 covenant_hash），但共用判卷调度逻辑（不重复造 cron）。

🔵 共用引擎 详细设计

反思流程（搬 LinDangAgent learning_reflector）

A三家各自反思
每家 AI 看自己的 hit_rate / 弱项板块 / 弱项方向：
  • "我在大涨股上 hit_rate 33%，明显低于平均 53%"
  • "我和另两家分歧时，我对的多还是错的多"
  • "debate 后我改观点的频率 / 改后准确率"
输出 self_proposals[]：建议改 covenant 第 X 条 / summary prompt / 个人 system prompt。
B跨家互审（圆桌特色）
每家审视另两家的 self_proposals：
  • Claude 看 Gemini 的 proposals，挑 evidence 不足 / sample bias / 风险未识别
  • Gemini 挑 Codex 的，Codex 挑 Claude 的（轮换）
  • 每条 proposal 标 pass / doubt / reject
C仲裁（仅当有 doubt 时）
Opus 4.7 充当仲裁者，看原 proposal + 质疑 + 答辩，做最终 adopt/reject 决策。
全部 pass 的 proposals 跳过仲裁直接进 staging。

Staging 与采纳分级

变更类型Staging 文件采纳条件
covenant 编辑 ~/.arena-research/staging/covenant.md 双轨 A/B 验证 +3% 自动采纳
profile 增量观察 ~/.arena-research/staging/profile.md 纯 append 到"增量观察"段，自动采纳（用户月度 review 整理）
summary prompt 编辑 ~/.arena-research/staging/research-rules.md 必须用户审批（diff + 理由 + 验证数据邮件提醒）
debate prompt 编辑 ~/.arena-research/staging/debate-template.md 同上，必须用户审批
三家分工 / 圆桌结构 不允许自动改 属于架构变更，不在 proposal 白名单

双轨 A/B 验证

验证集：

  • 从轨 1 quant_session.jsonl 抽 5 个 holdout（反思过程不参与）
  • 从轨 2 nat_session.jsonl 抽 5 个 holdout

跑两次圆桌：

  • 旧配置（生产 covenant + prompt）跑 holdout → 旧 hit_rate
  • 新配置（staging）跑 holdout → 新 hit_rate

采纳条件（必须全部满足）：

  1. 轨 1 holdout：新 hit_rate ≥ 旧 + 3%
  2. 轨 2 holdout：新 hit_rate ≥ 旧 + 3%
  3. 没有"某类样本断崖下跌 >15%"

任一轨退步即整体回退——这是天然的反偏科机制。改进只能让两轨都受益才落地。

共用引擎 Issue（3 条）

S1proposals 类型白名单
Opus 反思可能给"三家分工"/"debate 模式"等架构级改动。这些不在 proposal 白名单内会被过滤掉。
方案：白名单严格定义 4 种类型（covenant_edit / profile_obs / summary_prompt_edit / debate_prompt_edit），其他类型由 Opus 写入 ~/.arena-research/architectural_suggestions.md 供你季度 review。
S2A/B 验证 holdout 数量
5 个 holdout 太少（统计意义弱）；20 个又太贵（每次反思要重跑 20 × 3 家 × 2 轨 = 120 次圆桌）。
方案：早期用 5 + 5 验证，跑稳后升到 10 + 10。
S3proposal 失败后的 cooldown
某条 proposal A/B 没过。要永久 reject 还是 6 个月后再试？
方案：写到 ~/.arena-research/rejected_proposals.jsonl，每条带 reject_reason。Opus 反思时自动注入这份历史，避免反复提同样建议。
