当前方案 vs 修复后方案 — 逐项对比

左侧红色 = 当前代码行为，右侧绿色 = 修复后目标，中间箭头 = 变化方向

1. Gemini 多轮完成检测

Bug Gemini tool use 中间轮截断审查结果
Gemini 读文件时产生中间 type:"gemini" + tokens.total 但 content=""。当前代码第一次 turn-complete 就 resolve，拿到的是空或中间文本。

当前 transcript-tap.js:568

if (obj?.type === 'gemini' && obj.tokens && obj.tokens.total != null) { // content="" 也触发！ emitIfComplete(content); }
→

修复后

if (obj?.type === 'gemini' && obj.tokens && obj.tokens.total != null && typeof obj.content === 'string' && obj.content.trim().length > 0) { emitIfComplete(content); }

2. 审查判定解析

设计缺陷 正则解析第一行 → 大量假 FLAG
副驾先分析再给结论（自然行为），第一行是分析文字 → 正则不匹配 → 全部变 FLAG → Claude 收到无意义"格式异常"提醒。

当前 parseReviewVerdict()

  • 取第一行，正则 /^(OK|FLAG|BLOCKER)/i
  • 匹配成功 → 彩色 banner (OK绿/FLAG黄/BLOCKER红)
  • 匹配失败 → 强制 FLAG + "格式异常: ..."
  • FLAG → 自动向 Claude PTY 推送提醒
→

修复后

  • parseReviewVerdict() 删除
  • Banner 降级为状态通知："Gemini 审查完成" + 原文前 150 字摘要
  • 不做 OK/FLAG/BLOCKER 判定
  • 不向 Claude PTY 推送任何内容

3. 审查结果传递给 Claude

设计缺陷 直接推送 PTY → 可能被吞 / 打断 Claude
当前 FLAG 时直接 pty.write("[副驾提醒 from Codex]...")。Claude 正忙时文字写入 stdin 被吞，空闲时又可能打断思路。

当前：直接推送

1副驾完成 → parseReviewVerdict
2FLAG → ipcRenderer.send('terminal-input', Claude PTY, "[副驾提醒...]")
3Claude 可能正忙 → 文字被吞
→

修复后：文件 + 下次注入

1副驾完成 → 原文写入 .arena/reviews/review-{ts}.md
2Hub 标记 pendingReviewId = "review-{ts}"
3下次用户给 Claude 发消息时，Hub 自动在消息前注入：
[系统提示] 副驾审查已完成，请读取 .arena/reviews/review-{ts}.md
4Claude 自然读取 → 自己判断 → 不会被打断

4. Claude 自触发审查

设计缺陷 文本标记 [REQUEST_REVIEW] 不可靠 + 结果传递链路脆弱
依赖 Claude 输出精确文本标记（软约束），Hub 检测后走同一个有 Bug 的 triggerReview 流程，结果再推送回 Claude PTY（可能丢失）。

当前：文本标记 → Hub 中继

1Claude 输出 [REQUEST_REVIEW]（可能不输出）
2TranscriptTap 检测文本标记
3Hub 发射 driver-auto-review → renderer
4renderer 调 triggerReview()（有 Gemini Bug）
5结果 → parseReviewVerdict（假 FLAG）
6FLAG 推送到 Claude PTY（可能被吞）
6 步链路，每一步都可能失败
→

修复后：MCP Tool 直接调用

1Claude 调用 mcp__hub__request_review({ focus: "..." })
2Hub MCP handler → executeReview()
3结构化结果 → Claude 的 tool_result
3 步链路，结构化通信，不可能丢失

5. 审查结果存储

Bug 单文件覆盖 + 布尔 pendingReview
连续两次 @review，第二次覆盖 review-result.md，Claude 读到第二次的结果。pendingReview 只是布尔值，不知道指向哪次审查。

当前

  • .arena/review-result.md 单文件 → 覆盖
  • meeting.pendingReview = true → 不知道是哪次
  • 审查结果只在 banner 显示 5-15s → 消失后无处查看
→

修复后

  • .arena/reviews/review-{timestamp}.md 多文件归档
  • meeting.pendingReviewId = "review-1714200000" 指向具体文件
  • 审查结果同时写入 timeline → 黑板永久可查
  • MCP 路径不设 pendingReview（同步返回，无需注入）

6. 副驾 prompt 发送 + 等待

当前 driver-request-review

  • 并行发送 ✓（A10 已改）
  • 等待：仅 turn-complete，无任何兜底
  • 副驾挂死 → Promise 永远不 resolve → Hub 卡住
→

修复后

  • 并行发送 ✓
  • 等待：turn-complete + 10min watchdog
  • watchdog 触发 → 返回 { status: "timeout" }，不悬挂

7. 核心逻辑架构

当前：逻辑散布在 renderer + main

  • triggerReview() 在 renderer（meeting-room.js:61-162）
  • driver-request-review IPC 在 main（main.js:506-535）
  • 上下文收集在 renderer，prompt 发送在 main
  • 两处都有审查逻辑，状态分散
→

修复后：公共 executeReview()

  • executeReview() 在 main process — 单一事实源
  • 负责：上下文 + prompt + 发送 + 等待 + 收结果 + 写文件 + timeline
  • @review IPC 调它，MCP Tool 也调它
  • 全局锁 _reviewInProgress 在此函数入口
  • renderer 只负责 Banner UI

8. MCP Tool（全新）

当前：不存在

  • Claude 只能通过 [REQUEST_REVIEW] 文本标记
  • Hub 检测标记 → 中继给 renderer → triggerReview
  • 6 步异步链路，每步都可能断
→

新增 MCP server

  • 新文件 core/mcp-review-server.js
  • Claude 启动时 --mcp-config 注册
  • 工具 request_review：Claude tool call → Hub 执行 → 结构化返回
  • 返回：{ reviewId, status, reviewers: { gemini: { status, summary, filePath }, codex: {...} } }
  • [REQUEST_REVIEW] 标记保留为降级方案

9. 并发安全

当前：无保护

  • 用户 @review + Claude 自触发可同时进行
  • 两份 prompt 写入同一个副驾 PTY → 输出混杂
  • review-result.md 互相覆盖
→

修复后：全局锁

  • executeReview() 入口检查 _reviewInProgress
  • @review 已有审查进行 → Banner 提示"审查进行中"
  • MCP Tool 已有审查进行 → 返回 { status: "busy" }
  • 同 turn 内 Claude 重复调 MCP → cooldown 拒绝
