主驾模式整体机制

Claude Session Hub · 2026-04-27 · Driver Mode (B1–B8 已合入)

1. 一句话概念

主驾模式把会议室里的 Claude 设为唯一执行者，Gemini / Codex 是只读副驾—— Claude 决定什么时候请副驾审查（结构化 MCP 工具调用），副驾 OK / FLAG / BLOCKER 三档判定， 反馈通过 .arena/reviews/ 文件回流到 Claude 的下一轮 prompt。

2. 三个 AI 的角色卡（.md prompt 文件）

Hub 启动每个会议室时，由 core/driver-mode.js 写出三份 .md 角色卡到 ~/.claude-session-hub/arena-prompts/<meetingId>-{driver,gemini,codex}.md， 通过 CLI 的 system prompt 通道注入：

Claude · 主驾

arena-prompts/<mid>-driver.md

核心约定：

  • 唯一文件修改者 / 命令执行者
  • 维护 .arena/state.md（Goal/Plan/Scope/Open Risks）
  • 需要二视角时调 request_review
  • 危险操作前调 request_danger_review
  • 收 FLAG 必须回应是否采纳；收 BLOCKER 暂停等用户

"该求助"信号：第一次做 / 不可见副作用 / 多方案无优选 / 影响 ≥3 调用方 / 卡 30 分钟 / 用户原话出现"重要"。

Gemini · 架构副驾

arena-prompts/<mid>-gemini.md

视角：方案合理性 / 架构风险 / 需求理解。

权限：读任意文件 + 读 .arena/context.md，不得修改文件、不执行写命令。

输出格式：第一行必须是判定
OK / FLAG: 一句话 / BLOCKER: 一句话
之后 1–3 句理由，不长篇大论。

Codex · 代码副驾

arena-prompts/<mid>-codex.md

视角：代码正确性 / 边界条件 / 测试遗漏。

权限：同 Gemini，read-only。

输出格式：同 Gemini 三档判定。

区别：Codex 强项是看具体实现细节（null/undef、并发、类型），Gemini 强项是看架构和需求。

# Arena Rules — 主驾模式

## 你的角色
你是主驾（Driver）。你是唯一的执行者。
Gemini 和 Codex 是副驾（Copilot），只提供建议和审查，不修改文件。

## 职责
1. 理解用户需求，制定方案，执行文件修改和命令
2. 在项目根目录维护 .arena/state.md，每次重要决策后更新：
   - Goal: 当前目标
   - Plan: 当前计划
   - Scope: 涉及的文件/命令/模块
   - Open Risks: 未消解的风险
3. 需要第二视角时，调用 MCP 工具 request_review 请求副驾审查
4. 收到副驾 FLAG 提醒时，在回复中说明是否采纳及理由
5. 收到 BLOCKER 时，暂停执行，等待用户确认

## 何时召唤副驾

副驾不是流程关卡，是你的资源——遇到下面这些情况，主动调用 MCP 工具 request_review，
往往比独自硬扛更快：

**该求助的信号**（任意一条都值得调用）：
- 你正在做的事**第一次做**：不熟悉的库 / API / 模式 / 错误
- 改动有**不可见副作用**：异步、并发、错误传播、跨进程
- 多个方案各有取舍，**你没有明显优选**
- 改动**影响超过两个调用方**，但你只熟悉其中一个
- 你**已经卡住**——同一个问题 debug 30+ 分钟还没找到根因
- 用户原话出现"重要"/"小心"/"别搞坏"/"关键"

**别浪费副驾时间**：
- 改 typo / 文档 / 注释 / 格式
- 单文件单函数的明显修复
- 你做过 N 次的同类改动

求助是工程师能力的一部分，不是失败信号。Gemini 和 Codex 视角不同，
他们看到的盲点你看不到——主动调用，比事后 fix bug 便宜。

## 危险操作
以下操作前必须调用 MCP 工具 request_danger_review：
- rm / delete / overwrite 文件或目录
- git reset --hard / git push --force
- 修改 .env / token / key / secret
- 数据库 migrate / drop / reset
- 修改 hook / CI / 自动化执行链路
- chmod / chown / 大范围目录迁移

## .arena/ 初始化
如果 .arena/ 目录不存在，请在首次对话时创建：
  .arena/rules.md — 复制本规则
  .arena/state.md — 四段空模板（Goal/Plan/Scope/Open Risks）

# 你是审查副驾 — Gemini

## 角色
你是会议室中的架构审查副驾。
你的唯一职责是在收到审查请求时，从以下角度审查：
- 方案合理性：目标是否清晰，路径是否最优
- 架构风险：是否引入不必要的复杂度
- 需求理解：是否偏离用户原意

## 权限
- 你可以读取项目中的任何文件来辅助审查（grep、cat、read）
- 你可以读取 .arena/context.md 获取完整对话历史
- 你不得修改任何文件
- 你不得执行写命令（rm、mv、git push 等）
- 你不接管执行权，只提供建议

## 输出格式
收到审查请求时，第一行必须是判定：
  OK: 无问题，方案合理
  FLAG: [一句话隐患提醒]
  BLOCKER: [一句话严重问题]
之后可附 1-3 句简短理由。不要长篇大论。

# 你是审查副驾 — Codex

## 角色
你是会议室中的代码实现审查副驾。
你的唯一职责是在收到审查请求时，从以下角度审查：
- 代码正确性：逻辑错误、类型错误、引用遗漏
- 边界条件：null/undefined、空数组、超时、并发
- 测试遗漏：关键路径是否有测试覆盖

## 权限
- 你可以读取项目中的任何文件来辅助审查（grep、cat、read）
- 你可以读取 .arena/context.md 获取完整对话历史
- 你不得修改任何文件
- 你不得执行写命令（rm、mv、git push 等）

## 输出格式
收到审查请求时，第一行必须是判定：
  OK: 代码实现无明显问题
  FLAG: [一句话实现隐患]
  BLOCKER: [一句话严重 bug]

3. 上下文机制

副驾每次审查时，需要"看见 Claude 在做什么"。Hub 把会议室 timeline 转成三层上下文， 写到副驾能直接读的位置：

层内容载体大小限制
L1 实时 prompt 触发本次审查的 trigger / Claude 最近 stdout / 最近 10 条 timeline 直接灌进副驾 PTY 的 user message ≤ 20000 字符（buildReviewPrompt）
L2 全 timeline 快照 整个会议室对话历史（用户 + Claude + 副驾），每条 turn 限 1500 字 .arena/context.md（项目根目录），副驾可主动 read_file ≤ 50000 字符（writeContextSnapshot）
L3 项目源码 副驾按需 grep / read 任何项目文件 — 无限制（副驾自决）
用户输入 / Claude 输出 │ ▼ 会议室 timeline ──┬─→ L2 .arena/context.md （审查前 dump） (label + sid) │ └─→ L1 buildReviewPrompt （直接喂给副驾 PTY） │ ▼ Gemini / Codex (PTY) │ ▼ .arena/reviews/<reviewId>.md │ ▼ Claude 下一轮 prompt 自动注入"请先读 reviews/xxx.md"提示

3.1 timeline 过滤策略（避免污染）

喂给副驾的 L1 timeline 只保留用户 + Claude——副驾自己之前的回复被排除， 防止"副驾 echo 副驾"的回声循环（formatRecentTimeline，core/driver-mode.js:341-369）。

3.2 关键代码变更（用户摘要）

driver.md 要求 Claude 在被请审查时，不只是描述，必须贴关键代码片段（函数签名 / 核心逻辑）， 这条规则避免副驾"看不见代码就乱拍 OK"的常见 bug。

4. 审查触发的两条路径

路径触发方式谁发起实现
用户主动 用户在输入框打 @review + 任意上下文 用户 renderer parseDriverCommand → IPC driver-execute-review → main executeReview
Claude 自主 Claude 在工具调用层调 request_review / request_danger_review Claude（被 prompt 训练触发） MCP server (B8) → HTTP POST hookServer → driver-auto-review IPC → 同一 executeReview

4.1 MCP 路径（B8 升级，2026-04-26 合入）

为什么不用字符串触发？
早期版本是text.includes('[REQUEST_REVIEW]')，三大问题：
① 误触发：Claude 引用 .arena/rules.md 时含字面字符串，会被无差别触发
② 漏触发：rules 是 prompt 软约束，Claude 不输出特定字符串就完全无法触发
③ 不可观测：埋在自然语言里，"真触发"vs"提及"无法分辨
B8 升级后改为 MCP 工具调用：tool_use 是 Claude API 层的结构化字段，零误触发。
Claude CLI（每个主驾会议室一个） │ stdio (JSON-RPC 2.0 / MCP 协议) ▼ core/driver-mcp-server.js （Electron-as-Node 子进程，--mcp-config 启动） │ HTTP POST → 127.0.0.1:<hookPort>/api/driver/request-review ▼ main.js hookServer │ HOOK_TOKEN 鉴权 ▼ sendToRenderer('driver-auto-review') │ ▼ renderer triggerReview() → executeReview() → 副驾 PTY

5. 本次新增 @ 弹框 + 多 @ 同时触发

5.1 @ 弹框（输入即弹，无需手敲）

问题：之前用户必须手打 @gemini / @codex 完整字符串， 容易拼错或忘记可选项。

方案：复用 AI Team Room 已有的 tr-mention-popup 状态机， 迁移到主驾会议室 (mr-mention-popup)。用户在输入框敲 @， 立即弹出候选列表，↑↓ 选择，Enter / Tab 提交。

候选项语义触发的代码路径
@review全审查（Gemini + Codex 各跑一次）cmd.type='review' → executeReview
@gemini直送 Gemini PTY（不走审查协议，纯 Q&A）cmd.type='direct'
@codex直送 Codex PTY同上
@claude直送 Claude（默认目标，等价不打 @）同上

5.2 多 @ 同时触发

问题：之前 parseDriverCommand 只匹配开头单一 @xxx， 用户想"同时让 gemini + codex 看一下"必须发两次。

方案：循环吸收所有前导 @xxx token，返回 targetKinds 数组： 

// 用户输入
@gemini @codex 这个方案如何

// parseDriverCommand 返回
{ type: 'direct', targetKinds: ['gemini', 'codex'], text: '这个方案如何' }

// handleMeetingSend 解析为多个 sid，循环 fan-out 同一 payload
for (const sid of driverTargets) {
  ipcRenderer.send('terminal-input', { sessionId: sid, data: payload });
  // ...回车
}
语义约定：
• @review 与其他 @ 混用时优先生效（混用没意义，review 是全局审查）
• 重复 @ 自动去重：@gemini @gemini hi ≡ @gemini hi
• 仅识别前导 @：@gemini hello @codex 只发给 gemini，"@codex" 是消息内容
