@review 上下文机制详解

用一个完整场景说明"哪些信息发给副驾、有没有重复"

场景设定

用户在主驾模式下让 Claude 修 bug，过程中做了一次 @review，之后继续修复，然后再次 @review：

#1用户login 接口返回 500，帮我修一下
#2Claude找到问题了，auth.js:42 的 token 校验逻辑有 null 判断缺失，我来修...
#3Claude已修复 auth.js，改了 3 处：1) token null check 2) 错误码从 500 改为 401 3) 加了日志
#4用户@review 检查下这个修复
#5GeminiOK: 修复合理，null check 覆盖了主要路径
#6CodexFLAG: 缺少 token 过期的边界测试
#7审查结果[OK] Gemini / [FLAG] Codex: 缺少边界测试
#8用户补一下 token 过期的测试
#9Claude已添加 3 个测试用例：expired token、malformed token、empty token，全部通过
#10用户@review 再检查一遍 ← 我们关注这次

第二次 @review 时，系统里存在的三套上下文机制

机制 1: Phase A 增量上下文 (syncContext)

增量上下文 — 给每个 validTarget 计算的
handleMeetingSend 进入后，Phase A 遍历 validTargets 做增量计算。

但在 driverMode 下，validTargets = 全部 subSessions（Claude+Gemini+Codex）
Phase A 会给 3 个 target 各算一份增量上下文。

然而 @review 在 Step 4 early return，contextBySid 从未被读取。

就算被读取，它算的是"该 session 自上次收到消息后，timeline 中产生的新 turn"
—— 这和 triggerReview 里自己去拿的 recentTimeline 内容高度重叠。
结论：@review 路径下 Phase A 是 纯浪费 — 计算了但没人用。 即使将来用上，也和 triggerReview 自己构建的上下文 重复。
