问题 1: 副驾完成检测机制

Hub @review vs cli-caller — 如何知道"副驾说完了"？

三种 CLI 的完成信号完全不同

CLI Hub @review（TranscriptTap） cli-caller（一次性进程）
Claude Stop hook
Claude CLI 内置 hooks 机制：当 agent loop 结束时触发 Stop hook → Hub 的 session-hub-hook.py POST 到 Hub → Hub 调用 ClaudeTap.notifyStop() → 读 transcript JSONL 取最后一轮 assistant 文本 → 发射 turn-complete
可靠性：高。Hook 是 Claude CLI 原生机制。
 进程退出
printf | claude -p → 进程 stdout 就是完整输出 → 进程退出 = 完成
可靠性：最高。进程退出不可能误判。
Codex rollout JSONL 文件监听
Codex CLI 写入 ~/.codex/sessions/YYYY/MM/DD/rollout-*.jsonl
Hub 的 CodexTap 用 JsonlTail（fs.watch + 增量读取）监听该文件
当出现 {"type":"event_msg","payload":{"type":"task_complete"}} 事件时 → 取 payload.last_agent_message → 发射 turn-complete
可靠性：高。task_complete 是 Codex 结构化事件。
 JSONL stdout 解析
printf | codex exec - --json → stdout 是 JSONL 事件流
进程退出后解析 item.completed → 取 item.text
可靠性：最高。结构化 + 进程退出。
Gemini JSONL/JSON 文件监听
Gemini CLI 写入 ~/.gemini/chats/session-*.jsonl（0.39+）或 .json（0.38）
Hub 的 GeminiTap：
- JSONL 模式：JsonlTail 监听，当出现 {"type":"gemini","tokens":{"total":N}} 时取 content → 发射 turn-complete
- JSON 模式：轮询 mtime，稳定 400ms 后读全文取最后一条 type:"gemini"
可靠性：中。Gemini 没有显式 "完成" 事件，靠 tokens.total 出现推断。如果 Gemini 多轮输出（tool use），每轮都会触发一次。
 JSON stdout
printf | gemini --output-format json -y → stdout 是 JSON
进程退出后取 .response 字段
可靠性：最高。进程退出 = 完成。
核心差异：cli-caller 用一次性进程——进程退出 = 100% 完成，不可能误判。
Hub 用活 PTY 会话——必须通过文件监听推断完成。Claude 最可靠（有 Stop hook），Codex 次之（有 task_complete 事件），Gemini 最弱（靠 tokens 出现推断）。

但活 PTY 的优势是：副驾保持完整会话状态 + 可以自行读项目文件 + 后续可追问。cli-caller 没有这些。

实际风险评估

场景 会发生什么 风险
Codex 在 tool use 中间 不会误触发。task_complete 只在整个任务完成时发 无风险
Gemini 多轮 tool use 每轮 tokens.total 出现都触发 turn-complete，但 text === lastText 去重可过滤大部分 低风险：可能在中间轮触发，但通常最后一轮的 text 最长会覆盖之前的
Claude Stop hook 没触发 @review 的 Promise 永远不 resolve（D1 已去掉超时） 用户需手动关 banner。但 Claude 的 hook 可靠性极高
副驾 PTY 崩溃/卡死 同上，Promise 永远不 resolve 罕见。用户可手动关 banner 或关闭副驾 session

问题 2: "Claude 正忙" 怎么处理

cli-caller 的做法 + Hub 的可选方案

cli-caller 是怎么做的

1Claude 自己是编排者，主动发起 CLI 调用
↓
2用 Bash run_in_background: true 后台启动 Gemini/Codex/DeepSeek
↓
3Claude 继续做其他事（或等待）
↓
4后台进程完成 → Claude 主动读取输出（BashOutput / Read）
↓
5Claude 自己综合所有输出 → 生成汇总表格
关键：cli-caller 中 不存在"打断 Claude"的问题，因为 Claude 自己就是编排者，它主动去读结果，而非被动接收推送。

Hub @review 的差异

1Hub 是编排者，用户触发 @review
↓
2Hub 向副驾 PTY 写入 prompt → 等 turn-complete
↓
3副驾完成 → Hub 收到结果
↓
4Hub 向 Claude PTY 推送审查反馈文本
Step 4 是"推送"——如果 Claude 正在执行任务，这段文字会插入到 Claude 的输入流。

Claude 正忙时的实际行为

场景 A: Claude 空闲（最常见）

用户 @review → Claude 刚完成操作 → 在等用户输入
Hub 推送 [审查反馈 from Gemini] ...
Claude 看到 → 立即处理 → 正常工作

无问题

场景 B: Claude 正忙（少见但存在）

用户发了 @review 但 Claude 还在执行上一轮
Hub 推送 [审查反馈 from Codex] ...
文本写入 PTY stdin → 被 Claude CLI 的输入缓冲收到
但 Claude agent loop 正在执行工具调用 → 这段文字被吞掉（不会出现在 Claude 的对话上下文中）

反馈丢失

三种解法

解法 A: 不主动推送，只写 timeline（最简）

1副驾完成 → banner 展示原文摘要
2原文写入 timeline 永久保存
3不向 Claude PTY 推送任何内容
4用户自己看 banner / timeline 后决定是否告诉 Claude
零干扰 零丢失 需要用户手动传达

解法 B: 检查 Claude 状态再推送

1副驾完成 → 检查 Claude 的 session status
2status === 'idle' → 推送反馈到 Claude PTY
3status !== 'idle' → 暂存到队列，等 Claude 空闲后推送
不丢失 不打断 需要实现"空闲检测 + 队列"（~30 行）
空闲检测：Claude 的 turn-complete 事件已有 → 上次 turn-complete 后无新输入 = 空闲

解法 C: 借鉴 cli-caller — Claude 主动拉取

1副驾完成 → 结果写入 .arena/review-result.md 文件
2Claude 的 system prompt（rules.md）中有规则："每轮操作前检查 .arena/review-result.md 是否有新审查反馈"
3Claude 在下次 agent loop 开始时自行读取 → 自行处理
零干扰 Claude 主动 依赖 Claude 遵守 rules.md 规则 不保证及时性

综合对比

A: 不推送 B: 状态检查+队列 C: Claude 主动拉
Claude 忙时 无干扰 排队等空闲 无干扰
反馈丢失 不丢（在 timeline） 不丢（队列保证） 取决于 Claude 是否读文件
自动闭环 无（用户手动传达） 有（空闲后自动推送） 半自动（Claude 主动拉但不保证）
实现复杂度 最简（删代码） 中（~30 行） 中（文件写入 + rules 改动）
与 cli-caller 一致性 不一致 部分一致 最一致（Claude 主动读结果）
我的建议：当前阶段用 A（不推送） 最安全——你在场手动操作，自己看 banner 就够了。
后续如果需要自动闭环，升级到 B（状态检查+队列），工程量小且可靠。
C 方案有创意但依赖 Claude 遵守软约束，不如 B 的硬保证。
