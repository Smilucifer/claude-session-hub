三、Codex 独家发现（中置信度）

问题严重度采纳建议

review storm 防护：Claude 连续/递归调 MCP 审查 高 采纳——同 turn 内不允许重复调用（锁 + cooldown）
统一 ReviewResult 数据结构 高 采纳——作为核心模型

四、最终决策

问题 1：公共逻辑

决策：方案 a — 提取 executeReview() 公共函数，入口分离。
三路共识

问题 2：Banner

决策：保留 — 降级为状态通知（"审查完成 / 部分失败"），不做判定颜色。
三路共识

问题 3：MCP 返回内容

决策：结构化摘要 + 文件路径（采纳 Gemini+Codex 建议）
返回 { reviewId, status, reviewers: { gemini: { status, summary(500字), filePath }, codex: {...} } }
Claude 需要细节时自己 read 文件。
2:1 采纳多数

问题 4：并发控制

决策：全局锁 + 请求去重
Phase 1 先用全局 _reviewInProgress（简单可靠），后续可升级到 Codex 建议的细粒度锁。
三路共识加锁

兜底超时

决策：保留 10 分钟 watchdog（采纳 Gemini+Codex 建议）
不做 idle timeout，但 10min 绝对上限防悬挂。
2:1

MCP 同步/异步

决策：Phase 1 同步（Claude tool call 自然阻塞，实现简单）
审查通常 20-60s，Claude 等得起。超过 10min watchdog 兜底返回 partial。
Phase 2 可升级两段式。
Claude 建议 + 实用性优先

MCP 触发 vs @review 的去重

决策：MCP 路径不设 pendingReview（采纳 Codex 发现）
MCP 同步返回结果给 Claude，不再通过 pendingReview 注入。只有 @review 路径才走文件注入。
Codex 独家发现
