02 何时触发协作

触发方式 A
用户显式触发
@review 全副驾审查
@gemini 问题 单独问 Gemini
@codex 问题 单独问 Codex
用户在会议室输入框中输入 @ 命令。
Hub 解析命令，将审查 prompt 发送到对应副驾。
触发方式 B
Claude 主动触发
[REQUEST_REVIEW]
Claude 在输出中包含此标记时，Hub 自动触发副驾审查。

建议触发场景（写入 rules.md）：
• 跨 3+ 文件修改
• 引入新依赖/新模块
• 架构变更或重构
• 存在未消解的 Open Risks
• Claude 自评不确定
触发方式 C
危险命令触发
[DANGER_REVIEW]
Claude 在执行危险操作前，按 rules.md 要求输出此标记。

危险操作清单（硬编码在 rules.md）：
• rm / delete / overwrite
• git reset --hard / push --force
• 修改 .env / token / secret
• db migrate / drop / reset
• 修改 hook / CI 链路

仅副驾 BLOCKER 时才需用户确认
Phase 1 约束：危险命令检测是"软约束"——通过 rules.md 指示 Claude 自行判断并输出标记。 Hub 没有 PreToolUse hook 做硬拦截。如果 Claude 忽略 rules.md 直接执行，Hub 无法阻止。 这是 Phase 2 的增强方向。

05 各 AI 的"人设"文件差异

Hub 在创建会议室时，为三个 AI 分别生成不同的 system prompt 文件，注入各自的 CLI。以下是三个文件的完整内容对比。

Claude 主驾 system prompt（via --append-system-prompt-file）
# Arena Rules — 主驾模式 ## 你的角色 你是主驾（Driver）。你是唯一的执行者。 Gemini 和 Codex 是副驾（Copilot），只提供建议和审查，不修改文件。 ## 职责 1. 理解用户需求，制定方案，执行文件修改和命令 2. 在项目根目录维护 .arena/state.md，每次重要决策后更新： - Goal: 当前目标 - Plan: 当前计划 - Scope: 涉及的文件/命令/模块 - Open Risks: 未消解的风险 3. 需要第二视角时，输出 [REQUEST_REVIEW] 请求副驾审查 4. 收到副驾 FLAG 提醒时，在回复中说明是否采纳及理由 5. 收到 BLOCKER 时，暂停执行，等待用户确认 ## 何时请求审查 以下场景输出 [REQUEST_REVIEW]，并在标记后附带简短说明： - 跨 3 个以上文件的修改 - 引入新依赖、新模块、新抽象 - 架构变更或跨文件重构 - 存在未消解的 Open Risks - 你对方案不确定 格式示例： [REQUEST_REVIEW] 正在重构认证模块，涉及 3 个文件，不确定 OAuth 回调的 错误处理是否完整。请重点看 handleCallback() 函数。 ## 危险操作 以下操作前必须输出 [DANGER_REVIEW] 并附带说明： - rm / delete / overwrite 文件或目录 - git reset --hard / git push --force - 修改 .env / token / key / secret - 数据库 migrate / drop / reset - 修改 hook / CI / 自动化执行链路 - chmod / chown / 大范围目录迁移 ## .arena/state.md 初始化 如果 .arena/ 目录不存在，请在首次对话时创建： .arena/rules.md — 复制本规则 .arena/state.md — 四段空模板（Goal/Plan/Scope/Open Risks） 创建后回复：主驾模式已初始化。
Gemini 副驾 system prompt（via GEMINI_SYSTEM_MD env）

08 与 multi-reviewer (cli-caller) 的对比借鉴

上下文传递方式对比
Multi-Review (cli-caller Part 6) Driver Mode 审查
上下文内容 Claude 用 Read 工具读取所有改动文件源码，拼成代码块，写入 system prompt 文件 Claude 最后一轮输出（含代码变更）+ Claude 自述摘要 + timeline 近期对话
传递方式 system prompt 临时文件 + stdin 极简触发 直接写入副驾 PTY stdin（一段完整 prompt）
副驾角色区分 同一份 system prompt，五个审查维度 Gemini 侧重架构/需求，Codex 侧重实现/边界（差异化角色 prompt）
输出格式 表格：文件 | 行号 | 问题 | 严重度 单行：OK / FLAG / BLOCKER: 理由
调用方式 spawn 临时 CLI 进程，跑完即退 复用会议室内已有的 PTY 会话
借鉴点：
1. Multi-review 的核心优势是传递真实源码而非摘要——副驾能看到具体代码。
   Driver Mode 的 Claude 最后一轮输出通常也包含代码 diff，但如果 Claude 只是描述了改动而没贴代码，副驾信息就不完整。
   → 改进：Claude rules.md 中要求触发审查时"贴出涉及的关键代码变更"。
2. Multi-review 的 system prompt 包含明确的审查维度清单——不是泛泛的"请审查"。
   → 已借鉴：Gemini/Codex 的角色 prompt 各有明确审查维度。
3. stdin 大小控制（<5000 字符）——driver mode 的审查 prompt 也应做截断保护。

11 黑板 Timeline 作为副驾上下文

核心思路：黑板 = 最好的上下文源

会议室 timeline 记录了所有对话（用户 + Claude + 副驾）。这是最完整的上下文。 与其只给副驾 Claude 的最后一轮输出，不如把 timeline 近期内容也带上。

层级 机制 何时生效 Phase 1?
