审查判定机制分析

Hub @review 的判定解析 vs cli-caller 多方审查，风险点对比

一、Hub @review 当前的判定解析

parseReviewVerdict() 逻辑

输入：副驾的完整输出文本（turn-complete 捕获的 text）
  ↓
取第一行：text.trim().split('\n')[0]
  ↓
正则匹配：/^(OK|FLAG|BLOCKER)\s*[:：]\s*(.+)/i
  ↓
匹配成功 → { verdict: 'OK', reason: '...' }
匹配失败 → { verdict: 'FLAG', reason: '副驾输出格式异常: ' + 第一行前150字 }

你担心的问题：副驾没按格式输出

副驾实际输出第一行 匹配结果 判定 是否正确
OK: 修复合理，覆盖了主要路径 匹配 OK 正确
ok：没有发现问题 匹配（/i 忽略大小写，[:：] 支持中文冒号） OK 正确
FLAG: 缺少边界测试 匹配 FLAG 正确
让我先看看代码... 不匹配 FLAG + "格式异常: 让我先看看代码..." 误报 副驾还没给判定就被截断
## 审查报告 不匹配 FLAG + "格式异常: ## 审查报告" 误报 副驾用 markdown 格式
I've reviewed the code. OK: looks good 不匹配（OK 不在行首） FLAG + "格式异常: I've reviewed..." 误报 判定在中间位置
经过分析，我认为这个修复是合理的。 不匹配 FLAG + "格式异常: 经过分析..." 误报 副驾表达了 OK 意思但没用关键词
BLOCKER: 严重安全漏洞 匹配 BLOCKER 正确但少见
核心风险：AI 模型（尤其是 Codex/GPT-5.5）经常在给出结论前先做分析。如果第一行是分析文字而非判定关键词，所有非标格式一律变成 FLAG。

实际后果：
  • 副驾认为没问题，但因为开头说了句分析 → 被判 FLAG → Claude 收到无意义的"格式异常"提醒
  • FLAG 提醒发给 Claude 后，Claude 看到的是 [副驾提醒 from Codex 1] 副驾输出格式异常: 经过审查，代码修复... → Claude 困惑
  • 用户在 banner 上看到一片黄色 FLAG，以为有问题，切 tab 一看发现副驾说的是"没问题"

二、cli-caller 多方审查的机制

整体架构对比

Hub @review（当前）

用户输入 @review
↓
Hub 构建 prompt → 写入副驾 PTY
↓
副驾在 PTY 中自由输出
↓
TranscriptTap 捕获 turn-complete
↓
正则解析第一行 → OK/FLAG/BLOCKER
↓
渲染 banner + 提醒 Claude

关键特征：
  • 副驾是活着的 PTY 会话，有完整项目上下文
  • 副驾可以自行 read/grep 文件
  • 输出是自由文本，用正则硬解析
  • 判定提取依赖格式遵从

cli-caller 多方审查

Claude 自己发起审查
↓
读取代码 → 写入 system prompt 文件
↓
printf | gemini --output-format json printf | codex exec - --json
↓
进程退出，stdout 收完
↓
Claude 自己阅读全部输出 → 综合判断
↓
Claude 生成汇总表格

关键特征：
  • CLI 是一次性进程（无 PTY、无会话状态）
  • 代码内容通过 system prompt 文件喂入
  • Gemini 返回 JSON（.response 字段）
  • Codex 返回 JSONL 事件流（item.text）
  • 判定由 Claude 自己综合，不靠正则

关键差异详解

维度 Hub @review cli-caller
输出捕获 TranscriptTap 监听 PTY 的 turn-complete 事件 → 获取副驾最后一轮完整文本 CLI 进程退出后读 stdout：
Gemini → JSON {response:"..."}
Codex → JSONL 事件流 {type:"item",item:{text:"..."}}
判定提取 正则解析第一行：/^(OK|FLAG|BLOCKER)/i
副驾不遵守格式 → 误判 FLAG		 Claude 自己读全文，理解语义后综合判断
不依赖任何格式约定
格式容错 零容错。第一行不是关键词就 FLAG 完全容错。Claude 能理解 "看起来没问题" = OK
上下文能力 副驾是活 PTY，可自行 read/grep 项目文件 CLI 一次性，代码必须全部喂入 system prompt
对话连续性 副驾保持完整会话状态，后续可追问 进程结束即丢失，无法追问
等待方式 turn-complete 事件（异步，无超时） 进程退出（同步 Bash timeout）
结果汇总 Hub 机械化 banner（OK 绿 / FLAG 黄 / BLOCKER 红） Claude 生成 markdown 表格（高/中置信度分类，多路共识标注）

三、优化方案

方案 A：增强正则（修补当前逻辑）

当前：只看第一行
改为：扫描前 5 行，找到第一个匹配 OK/FLAG/BLOCKER 的行

额外兜底：如果前 5 行都没匹配到，扫描全文找关键词：
  - 出现 "没有问题"/"合理"/"no issue"/"looks good" → OK
  - 出现 "建议"/"注意"/"suggest"/"consider" → FLAG
  - 出现 "严重"/"危险"/"must fix"/"critical" → BLOCKER
  - 都没有 → FLAG（保守兜底）
优点 改动小，向后兼容
缺点 仍然是启发式猜测，永远有漏网之鱼

方案 B：放弃机械解析，让 Claude 综合判断（借鉴 cli-caller）

副驾输出回来后，不做正则解析
而是：把副驾原文直接展示在 banner 中（截取前 200 字）
同时把原文发给 Claude，让 Claude 自己判断是否有问题

流程变为：
  副驾输出 → 直接展示原文摘要到 banner
           → 原文追加到 timeline
           → 如果用户开启了"Claude 自动处理"，则 Claude 自行消化
优点 彻底消除格式依赖，副驾自由发挥
缺点 没有结构化的 OK/FLAG/BLOCKER 颜色编码
缺点 Claude 需要额外处理副驾输出（多一步）

方案 C：混合 — 宽松解析 + 原文兜底

1. 扫描全文（不只是第一行）找 OK/FLAG/BLOCKER 关键词
2. 找到 → 按关键词着色 banner + 显示 reason
3. 找不到 → 不当作 FLAG，而是显示为"INFO"灰色，展示原文摘要
4. INFO 判定不触发 Claude 提醒（不打扰主驾）
5. 原文始终写入 timeline（永久可查）
优点 格式正确时享受结构化颜色；格式异常时不误报
优点 向后兼容，副驾遵守格式时体验不变
优点 不强依赖任何格式，消除"假 FLAG"干扰

四、决策

方案改动量建议
A: 增强正则（扫5行+关键词兜底） ~20 行 治标不治本
B: 放弃解析，原文展示 + Claude 判断 ~40 行 最干净但失去结构化 banner
C: 宽松扫全文 + INFO 兜底 ~30 行 推荐 — 兼顾结构化与容错
