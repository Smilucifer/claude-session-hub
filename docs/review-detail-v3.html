@review 自动闭环 — 每一步实际做了什么

具体场景 + 实际 prompt + 实际结果展示

场景：Claude 刚修完 auth.js 的 bug（3 处改动），用户输入 @review 检查下这个修复

Step 1: 构建上下文

1a
自动 写 .arena/context.md 快照
把完整 timeline（全部角色）写入磁盘文件。副驾的 prompt 会告知此文件存在，如果注入的上下文不够用，副驾可自行 read .arena/context.md 获取完整历史。
1b
自动 获取 Claude 最近一轮完整输出
调用 get-last-assistant-text 从 TranscriptTap 缓存读取。
得到的内容（示例）：
已修复 auth.js，改了 3 处： 1) auth.js:42 — 添加 token null check：`if (!token) return res.status(401).json({error: 'missing token'})` 2) auth.js:58 — 错误码从 500 改为 401 3) auth.js:61 — 添加 console.warn 日志
1c
自动 获取近期 Timeline（仅 Claude + 用户，D2 过滤后）
调用 driver-recent-timeline，最近 10 条，过滤掉 Gemini/Codex 输出。
得到的内容：
[会议室近期对话（Claude + 用户）] 【用户】login 接口返回 500，帮我修一下 【Claude】找到问题了，auth.js:42 的 token 校验逻辑有 null 判断缺失，我来修... 【Claude】已修复 auth.js，改了 3 处：1) token null check 2) 错误码从 500 改为 401 3) 加了日志 【用户】@review 检查下这个修复 ---

Step 2: 构建 Prompt（每个副驾不同）

2
自动 拼接审查 Prompt
把 1b + 1c + 用户文字 + 角色说明 + 格式要求拼成完整 prompt。Gemini 和 Codex 收到的角色说明不同：
发给 Gemini 的完整 Prompt
=== 审查请求 === 角色: 你是架构审查副驾（Gemini）。 请从方案/架构/需求理解角度审查。 触发: user-review --- 近期对话 --- 【用户】login 接口返回 500，帮我修一下 【Claude】找到问题了，auth.js:42... 【Claude】已修复 auth.js，改了 3 处... 【用户】@review 检查下这个修复 --- Claude 最近操作 --- 已修复 auth.js，改了 3 处： 1) auth.js:42 — 添加 token null check 2) auth.js:58 — 错误码从 500 改为 401 3) auth.js:61 — 添加 console.warn 日志 --- 用户补充 --- 检查下这个修复 如需更多上下文，可读取 .arena/context.md 或项目源码文件。 请用以下格式回复（第一行必须是判定）: OK|FLAG|BLOCKER: 一句话理由
发给 Codex 的完整 Prompt
=== 审查请求 === 角色: 你是代码实现审查副驾（Codex）。 请从代码正确性/边界条件/测试遗漏角度审查。 触发: user-review --- 近期对话 --- （同左，完全一样） --- Claude 最近操作 --- （同左，完全一样） --- 用户补充 --- 检查下这个修复 如需更多上下文，可读取 .arena/context.md 或项目源码文件。 请用以下格式回复（第一行必须是判定）: OK|FLAG|BLOCKER: 一句话理由

Step 3: 群发 + 等待

3a
自动 显示 "审查中..." Banner
终端区上方出现黄色横幅 审查中...，用户知道流程已启动。
3b
自动 并行写入两个副驾 PTY
Promise.all 同时向 Gemini PTY 和 Codex PTY 写入各自的 prompt + \r（Enter）。
延迟：Gemini 250ms + payload比例 / Codex 500ms + payload比例，但两者并行不累加。
3c
等待 监听 turn-complete 事件
TranscriptTap 监控每个副驾的 CLI 输出。当 CLI 回到输入提示符（turn 结束），触发 turn-complete 事件。
D1 简化后：无任何超时，完全等待副驾自然完成。Codex 思考 2 分钟也不会被打断。

用户体验：此时你可以切到 Gemini/Codex tab 实时观察它们的输出流。

Step 4: 解析判定

4
自动 解析 OK / FLAG / BLOCKER
取副驾输出的第一行，正则匹配 /^(OK|FLAG|BLOCKER)\s*[:：]\s*(.+)/i
匹配失败 → fallback 为 FLAG + 原文前 150 字符

实际副驾输出示例：
OK: 修复合理，null check 覆盖了主要路径，错误码改为 401 符合 HTTP 语义 详细分析： - token null check 在 validateToken 入口，覆盖了所有调用路径 ✓ - 500→401 改动正确，认证失败不应该是服务端错误 ✓ - 建议后续补充 token 过期的边界测试 ↑ 只解析第一行 "OK: ..."，后面的详细分析副驾自由发挥

Step 5: 展示结果

5a
自动 渲染结果 Banner
替换 "审查中..." 为彩色判定横幅：
Gemini 1 OK 修复合理，null check 覆盖了主要路径，错误码改为 401 符合 HTTP 语义
Codex 1 FLAG 缺少 token 过期场景的边界测试用例
自动消失规则：全 OK → 5s | 有 FLAG → 15s | 有 BLOCKER → 不消失，需手动操作
5b
自动 写入 Timeline 永久记录
审查结果写入会议 timeline，在黑板模式可永久查看：
**审查结果** [OK] Gemini 1: 修复合理，null check 覆盖了主要路径，错误码改为 401 符合 HTTP 语义 [FLAG] Codex 1: 缺少 token 过期场景的边界测试用例
5c
自动 FLAG → 向 Claude PTY 发送提醒
Codex 判定 FLAG → Hub 自动向 Claude 的 PTY 写入提醒文本：
[副驾提醒 from Codex 1] 缺少 token 过期场景的边界测试用例
Claude 看到后会自行决定是否处理。OK 判定不发提醒（不打扰主驾）。
BLOCKER 判定也发提醒，但 banner 不自动消失，等用户决定"覆盖继续"或"中止"。

对比：@review 一键 vs @gemini + @codex 手动

@review 检查下这个修复

  • 用户输入 1 条消息
  • Hub 自动构建含 Claude 上下文的审查 prompt
  • 自动群发两个副驾
  • 自动等待 + 解析判定
  • 自动展示 banner + 写 timeline
  • FLAG 自动提醒 Claude
  • 用户全程只需等结果
vs

手动 @gemini + @codex

  • 用户输入 2 条消息，每条都要手写上下文
  • 例如：@gemini Claude 刚改了 auth.js 3 处，帮我从架构角度审一下
  • 再输入：@codex 同上，帮我从代码层面查
  • 自己切 tab 看结果
  • 自己判断 OK / 有问题
  • 如有问题，自己手动告诉 Claude
  • 无 banner、无 timeline 记录、无自动提醒
核心区别：@review 的价值不在于"群发"（@gemini + @codex 也能做到），而在于自动构建上下文 + 结构化结果收集 + 自动闭环提醒。
如果你觉得手动提供上下文更精准、自动构建反而带来噪音，那 @gemini/@codex 手动模式更适合。
两者不冲突 — 日常快速检查用 @review，精细控制用 @gemini/@codex。
