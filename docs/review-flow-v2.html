@review 流程简化方案

基于用户反馈的修改方案，待决策确认后实施

D1: 去掉所有阻塞超时判断

D1 屏蔽全部超时判断，仅保留 turn-complete 自然结束 已部分完成
当前状态：idle timeout 已屏蔽，但 wall clock 120s 和 safety net 125s 仍在。

建议方案：全部去掉。副驾结束的唯一信号是 turn-complete 事件（TranscriptTap 检测到 CLI 回到输入提示符）。用户在场可自己观察。

效果：
  • Codex 思考 1m26s（如截图）不再被 FLAG
  • 副驾无论花多久都能完整回复
  • 极端情况（副驾卡死/永不回复）：用户手动关 banner 或切走

D2: 上下文投喂 — 只给 Claude+用户 的对话

D2 Timeline 过滤：副驾审查只看 Claude + 用户 的 10 条 待确认
当前问题：formatRecentTimeline() 返回最近 10 条，包含 Gemini/Codex 自己的输出。
副驾看到自己之前的输出 → 信息冗余 + token 浪费。

改动：formatRecentTimeline() 新增 filterKinds 参数，默认只取 sid === 'user' 或 kind === 'claude' 的 turn。
过滤示意 
Timeline 原始数据（最近 15 条）:
  #1  [用户]  "分析下这段代码"
  #2  [Claude] "我来看看... 发现了3个问题..."
  #3  [Gemini] "从架构角度看..."          ← 过滤掉
  #4  [Codex]  "代码层面的问题..."         ← 过滤掉
  #5  [用户]  "修一下问题2"
  #6  [Claude] "已修复，改了 main.js:42..."
  #7  [用户]  "@review 检查下"
  #8  [审查结果] "[OK] Gemini / [FLAG] Codex" ← 过滤掉（系统生成）

审查 Prompt 中实际注入的上下文:
  [用户]  "分析下这段代码"
  [Claude] "我来看看... 发现了3个问题..."
  [用户]  "修一下问题2"
  [Claude] "已修复，改了 main.js:42..."
  [用户]  "@review 检查下"

+ 提示副驾："如需更多上下文，可读取 .arena/context.md（完整对话历史）或项目源码"
D2b context.md 也做同样过滤 待确认
.arena/context.md（完整快照）同样过滤掉 Gemini/Codex turns。
副驾通过 prompt 提示知道"黑板有完整历史"，需要时可自行查 meeting-get-timeline（但副驾无 IPC 权限，只能读文件）。

或者：context.md 保留全量（作为副驾主动查阅的"完整档案"），只在 prompt 注入部分过滤。
建议：context.md 保留全量。prompt 注入过滤。两层信息递进。

D3: Claude 摘要开关 — 去掉轮询阻塞

D3 摘要开关开启时：发送指令但不等待，改为取 Claude 已有最新输出 待确认
当前问题：摘要开关开启时，向 Claude 发送摘要指令 → 轮询 10×2s = 最多 20s 阻塞 等 Claude 回复。

方案 A（推荐）：删除轮询。摘要开关开启时，直接把 get-last-assistant-text（Claude 最近一轮完整输出）作为摘要注入上下文。不主动向 Claude 发任何指令。
理由：Claude 刚完成操作，最后输出就是最好的"摘要"。无需额外让 Claude 花时间总结。

方案 B：保持发指令，但 fire-and-forget 不等回复。Claude 的摘要出现在 Claude tab 里供用户看，但不注入副驾 prompt。

摘要开关的新语义：开启 = 把 Claude 最近输出加入副驾上下文 | 关闭 = 不加

D4: Step 2 增量上下文的价值判断

D4 增量上下文对 @review 路径无价值，可安全跳过 待确认
增量上下文是什么：当 syncContext 开关打开时，普通消息发送前会计算"自上次发送以来对方产生的新 turn"，注入当前消息中，实现会话间同步感知。

对 @review 的影响：
  • @review 在 Step 4 就 early return 了，计算结果 contextBySid 完全没被使用
  • 增量上下文对普通消息有价值（"全部"发送时让每个 AI 看到对方回复），但审查走独立流程
结论：在 @review 路径中是纯浪费。应在 driverMode + @review 时跳过 Phase A。

对普通消息的价值：syncContext + 非审查消息时仍有用（例如 @gemini 直接提问时注入 Claude 上下文），保留。

D5: PTY 发送改并行

D5 副驾 Prompt 发送从串行改为并行 待确认
当前：for (sid of sids) { write + await sleep + write('\r') } — 串行，2 个副驾约 1-1.5s
改为：await Promise.all(sids.map(...)) — 并行写入各自 PTY
节省：~500-750ms
风险：无。各 PTY 独立进程，互不干扰。

简化前后对比

当前流程（16 步）

  1. doSend → handleMeetingSend
  2. Phase A 增量上下文 (5-20ms 浪费)
  3. Timeline 写入用户消息
  4. 解析 @review → PENDING banner
  5. 查找副驾 SID
  6. 写 context.md (全量含 Gemini/Codex)
  7. Claude 摘要轮询 (0-20s 阻塞)
  8. 获取 Claude 最近输出
  9. 获取近期 Timeline (含全部角色)
  10. 构建 Prompt
  11. driver-request-review IPC
  12. 串行发送 Prompt (500-1500ms)
  13. 等待 AI (turn-complete / 120s超时)
  14. 渲染 Banner
  15. 写入 Timeline
  16. FLAG → 提醒 Claude

简化后流程（12 步）

  1. doSend → handleMeetingSend
  2. — 跳过增量上下文 —
  3. Timeline 写入用户消息
  4. 解析 @review → PENDING banner
  5. 查找副驾 SID
  6. 写 context.md (全量保留)
  7. 摘要开关=ON → 直取 Claude 最近输出作为摘要（无轮询）
  8. 获取近期 Timeline (仅 Claude+用户)
  9. 构建 Prompt
  10. driver-request-review → 并行发送 Prompt
  11. 等待 AI (仅 turn-complete，无超时)
  12. 渲染 Banner + 写 Timeline + 提醒 Claude
