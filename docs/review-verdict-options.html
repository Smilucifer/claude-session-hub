审查判定方案对比

前提：采用方案 B（展示原文，不做机械解析）。问题变为：如何确定 OK / FLAG / BLOCKER？

思路 1: Claude 自己判断

思路 1 副驾原文直接发给 Claude，Claude 自己消化判断
1副驾完成输出 → TranscriptTap 捕获全文
2Banner 显示 "Gemini 已完成审查"（中性灰色，不判断对错）
3副驾原文摘要（前 300 字）写入 timeline 永久保存
4副驾原文 发送到 Claude PTY：[审查反馈 from Gemini]\n{全文}\n---
5Claude 自行阅读、自行决定是否需要行动（就像 cli-caller 一样）

Claude 收到的内容示例：

[审查反馈 from Gemini] 经过分析，auth.js 的修复覆盖了主要路径。token null check 放在 validateToken 入口是合理的。错误码从 500 改为 401 符合 HTTP 语义。 不过有一点值得注意：当 token 存在但已过期时，当前逻辑返回的是 generic 401，没有区分 "missing" 和 "expired" 两种情况。建议后续 补充过期场景的专门错误信息。 总体评价：修复合理，可以合入。 --- [审查反馈 from Codex] 审查了 auth.js 的 3 处改动。 1. auth.js:42 null check — 正确 2. auth.js:58 错误码 — 正确 3. auth.js:61 日志 — console.warn 在生产环境会被吞掉， 建议改用项目的 logger 模块 缺少测试：token 过期、malformed token 两个边界情况没有测试覆盖。 ---

Banner 展示效果：

Gemini 修复覆盖了主要路径...建议后续补充过期场景的专门错误信息。总体评价：修复合理，可以合入。
Codex console.warn 建议改用 logger 模块...缺少测试：token 过期、malformed token 边界情况。
优点
  • 零格式依赖，副驾完全自由发挥
  • Claude 理解语义，不会误判
  • 和 cli-caller 的模式一致，Claude 已熟悉
  • 最简实现 — 删掉 parseReviewVerdict，改为直接转发
代价
  • Banner 没有 OK/FLAG 颜色区分（全灰）
  • 用户需要自己读摘要或看 Claude 的反应来判断严重程度
  • 如果 Claude 正在执行任务，审查反馈会打断它

思路 2: 末行判定（比首行自然）

思路 2 Prompt 要求末行输出判定（分析→结论，符合 AI 自然思维）
1Prompt 改为："最后一行必须是判定：OK|FLAG|BLOCKER: 理由"
2副驾自然先分析、再下结论（不用强迫先说结论）
3解析时扫描 最后 3 行 找关键词（而非第一行）

副驾自然输出（不用刻意调整风格）：

经过分析，auth.js 的修复覆盖了主要路径。 token null check 在 validateToken 入口是合理的。 错误码从 500 改为 401 符合 HTTP 语义。 不过 console.warn 在生产环境会被吞掉，建议改用 logger。 缺少 token 过期的边界测试。 FLAG: console.warn 应改用 logger，且缺少 token 过期边界测试
优点
  • 符合 AI 自然表达：先分析后结论
  • 比首行格式遵从率高很多（AI 做完分析后倾向于总结）
  • 保留了结构化 OK/FLAG/BLOCKER 颜色编码
代价
  • 仍然依赖格式。如果末尾说 "以上是我的分析" 就又匹配不到
  • 比首行好，但不是 100% 可靠

思路 3: 综合 — 末行优先 + Claude 兜底

思路 3 末行快速提取 → 失败时 Claude 兜底判断 推荐
1Prompt 要求末行输出判定（思路 2 的改进）
2副驾完成 → 扫描最后 5 行找 OK|FLAG|BLOCKER

3a快速路径：直接用提取到的判定 → 彩色 banner（~0ms）

  • 兜底路径会向 Claude 发消息，如 Claude 正忙可能干扰

"打断 Claude" 问题的解法

思路 1 和 3 都会向 Claude PTY 发消息，但场景不同：

@review 触发时机：通常是 Claude 刚做完一轮操作，用户主动 @review → Claude 此时大概率空闲（在等用户输入）。
所以审查反馈发给 Claude 时，Claude 不太会"正忙"。

但如果 Claude 确实在忙（用户 @review 的同时 Claude 还在跑）：
→ 反馈文本会排在 Claude 的输入队列里，Claude 完成当前任务后自动看到
→ 不会丢失，只是延迟处理

结论：这不是真正的"打断"问题，更像是"排队"。
