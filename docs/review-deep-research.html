Q2: Gemini 多轮问题 — 找到了根本解法，不需要 debounce

发现：Gemini JSONL 中间轮 content 为空

实际查看 ~/.gemini/chats/session-*.jsonl 文件，发现 Gemini 多轮 tool use 时的 JSONL 模式：
// ---- 中间轮（tool use 发起后的中间状态）---- {"id":"85ce8ec6-...","type":"gemini","content":"","tokens":{"total":4563},"toolCalls":[...]} {"id":"85ce8ec6-...","type":"gemini","content":"","tokens":{"total":4563},"toolCalls":[...]} // ---- 最终轮（Gemini 完成所有 tool use 后的真正回复）---- {"id":"a1b2c3d4-...","type":"gemini","content":"OK: 修复合理，覆盖了主要路径...","tokens":{"total":8921}}
关键发现：中间轮的 content 为空字符串 ""，最终轮的 content 是真正的回复文本。
这不是偶然——Gemini CLI 在 tool use 过程中还没生成回复，所以 content 为空。只有全部 tool use 完成、Gemini 最终回复时才有真正 content。

修复方案：加一个 content 非空检查

当前代码（有 Bug）

// transcript-tap.js:568 if (obj?.type === 'gemini' && obj.tokens && obj.tokens.total != null) { // 中间轮也匹配！content="" 也触发 emitIfComplete(content); }

修复后

if (obj?.type === 'gemini' && obj.tokens && obj.tokens.total != null && typeof obj.content === 'string' && obj.content.trim().length > 0) { // 只有真正回复才触发 emitIfComplete(content); }

对比 debounce 方案

content 非空检查（官方信号） debounce 5s（之前提的）
原理 利用 Gemini JSONL 的数据结构特征 猜——等够久再取
额外延迟 0ms +5s
可靠性 确定性判断（content 有值 = 回复完成） 概率性（5s 够不够？tool use 间隔 > 5s 怎么办？）
改动量 1 行 ~15 行
结论：这就是你说的"更好的官方方案"。不需要 debounce，一行代码解决。
推荐立即实施

Q3: Claude 自编排审查 — 三种思路深度对比

核心矛盾

cli-caller 模式（Claude 直接编排）之所以鲁棒，是因为：
  • Claude 是调用者 → 它主动发起、主动等待、主动读结果
  • 进程退出 = 完成 → 不需要推断完成信号
  • 结果在 stdout → 不需要文件监听或事件中继

Hub @review 模式之所以脆弱，是因为：
  • Hub 是编排者 → Claude 只能被动接收推送
  • PTY 会话不退出 → 必须推断完成信号（TranscriptTap）
  • 结果在事件流 → 需要文件监听 + IPC 中继

问题：能否让 Claude 在 Hub PTY 中也像 cli-caller 一样"主动编排"？

思路 A: Claude 在 Hub PTY 中直接调 CLI（模仿 cli-caller）

// Claude 在 Hub 的 PTY 会话中执行： printf '%s' "请审查 auth.js 的修复" | gemini --output-format json -y // → 进程退出 → Claude 读 stdout → 得到结构化 JSON
可行性可行。Claude 在 PTY 中可以执行 Bash 命令
优点100% 复用 cli-caller 的模式；进程退出 = 完成；结构化输出
缺点 1需要完整的环境配置（代理、路径、system prompt 文件）—— cli-caller skill 里做了 10+ 步前置检查，Claude 在 Hub PTY 里不一定能全做对
缺点 2绕过了 Hub 的副驾管理——Hub 不知道 Claude 发起了审查，无法在 timeline/banner 中显示
缺点 3一次性 CLI 没有项目上下文（代码要全部通过 system prompt 塞进去），而 Hub 的活 PTY 副驾已经有完整项目环境
缺点 4并发限制——Claude 自己只能串行调，不能并行 Gemini+Codex

思路 B: MCP Tool — Hub 暴露工具给 Claude 调用

Hub 启动时注册一个 MCP server，Claude 的 session 通过 --mcp-config 连接。
Claude 直接用工具调用发起审查：
// Claude 的工具调用（自然融入 agent loop）： Tool: mcp__hub__request_review Input: { "focus": "auth.js 的 token null check 修复", "files": ["auth.js"], "style": "both" } Tool Result: { "status": "completed", "gemini": { "verdict": "OK", "full_text": "修复合理，null check 覆盖了主要路径..." }, "codex": { "verdict": "FLAG", "full_text": "缺少 token 过期边界测试..." } } // Claude 自己读结果、自己判断、自己决定下一步
可行性可行。Hub 已有 MCP 基础设施经验
优点 1结构化输入输出——无格式解析问题
优点 2Claude 自然阻塞在 tool call 上——不存在"Claude 正忙"问题
优点 3Hub 保持编排权——可以记 timeline、显示 banner、管理副驾
优点 4副驾在活 PTY 中有完整项目上下文
缺点需要构建 MCP server + 注册到 Claude session（~100 行新代码）

思路 C: 改良当前方案 — 文件触发 + 下次注入

保持当前 [REQUEST_REVIEW] 标记检测，但修复下游：
// Claude 输出 [REQUEST_REVIEW] → Hub 检测到 // Hub 触发审查 → 结果写入 .arena/review-result.md // Hub 标记 pendingReview = true // 下次 Claude 收到用户消息时，Hub 自动注入提示： [系统提示] 副驾审查已完成，请先读取 .arena/review-result.md // Claude 读文件、处理反馈
可行性可行且改动最小
优点Q1 方案已经设计好，只需加上 Claude 自触发的场景
缺点 1仍依赖文本标记 [REQUEST_REVIEW]（软约束）
缺点 2结果延迟——Claude 要等到下次交互才读到
缺点 3Claude 自触发后继续工作 → 审查结果可能来得太晚

三思路总结对比

A: Claude 直调 CLI B: MCP Tool C: 标记 + 文件注入
Claude 主动权 完全主动 完全主动（tool call） 被动等结果
格式可靠性 JSON stdout MCP 结构化返回 文本标记 + 文件
副驾上下文 无（一次性进程） 完整（活 PTY） 完整（活 PTY）
结果及时性 同步返回 同步返回（tool result） 延迟到下次交互
Hub 可见性 Hub 不知道 Hub 完全掌控 Hub 完全掌控
实现量 ~20 行（但配置复杂） ~100 行 ~30 行
与 cli-caller 一致性 最像 精神一致（Claude 主动调用 → 等结果 → 自己综合） 不像
