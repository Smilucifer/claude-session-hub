D. 架构审视 — 需要多方评审的问题

问题 1：@review（用户触发）和 MCP Tool（Claude 触发）应共享多少逻辑？

两者都要：构建上下文 → 发给副驾 → 等结果 → 展示/记录。
方案 a：提取公共函数 executeReview()，@review 和 MCP Tool 都调用它。
方案 b：分离——@review 走 renderer IPC，MCP Tool 走 main process 内部调用。逻辑有少量重复但解耦。

问题 2：去掉 parseReviewVerdict 后，Banner 还有存在价值吗？

没有颜色区分（全灰），没有结构化判定，只是原文摘要 → 本质上和 timeline 中的记录重复。
方案 a：保留 Banner 作为"即时提醒"（看一眼就知道审查完了），timeline 是永久记录。
方案 b：删除 Banner，结果只进 timeline + .arena/review-result.md。用户切 tab 看副驾原文或看黑板。

问题 3：MCP Tool 返回给 Claude 的应该是原文还是摘要？

副驾可能输出 2000+ 字的详细审查。全量返回给 Claude 吃 context？
方案 a：返回全文（Claude 自己决定取舍，最鲁棒）
方案 b：返回前 500 字 + 提示"完整内容见 .arena/review-result.md"（省 token）

问题 4：用户 @review 和 Claude MCP 审查可以同时进行吗？

如果用户 @review 的同时 Claude 也 MCP 调审查 → 两个审查同时发给副驾 → 副驾 PTY 收到两份 prompt → 混乱。
方案：加锁——同一时间只允许一个审查进行。第二个排队或拒绝。
